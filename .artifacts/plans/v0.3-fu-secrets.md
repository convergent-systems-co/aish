# Plan: v0.3-fu-secrets — Follow-up wave for the secrets engine

> Seven backlog issues against the v0.3-3 secrets engine. Three
> conceptual groups (taint, Linux backend, cross-engine wiring) cut
> ruthlessly into a **4-task MVP floor** and a **3-task deferral set**
> so the PR stays reviewable and stays inside the secrets-package
> boundary.

**Source:** GOALS.md §"Epic v0.3-3 — Identity & Secrets Engine"
**Parent epic (supervised):** #95
**Tasks shipping this PR:**
- **#96** Taint bit type system (MVP — command-level, not per-token)
- **#98** History exclusion via taint
- **#99** Pipe taint propagation (MVP — sticky-bit, not per-byte)
- **#101** Linux Secret Service backend via D-Bus
- **#105** Persona-bound secrets (labels filter)

  shipped in v0.3-3; this PR adds the persona-switch event + the
  signature assertion test)

- **#104** Atomic persona switch — SSH config, cloud profiles, kube
  context, git config. Cross-cuts files outside `domain:secrets`.
  Documented seam, follow-up filed.

**Domain:** `domain:secrets`
**Risk:** HIGH per GOALS.md. The new Linux backend lives behind the
existing `Backend` interface so the crypto surface does not change.

parser/history behavior is unaffected when no command in the pipeline
is tainted.

## Acceptance Criteria

### MVP-floor (shipping this PR)

#### #101 Linux Secret Service backend

- [ ] **`OpenLinuxBackend(prefix string, entropy []byte) (Backend, error)`**
      in a new `shell/internal/secrets/backend_linux.go` behind
      `//go:build linux`.
- [ ] When no D-Bus session bus is available OR the
      `org.freedesktop.secrets` service is not on the bus, the
      constructor returns a wrapped `ErrUnsupported` — same posture as
      the macOS backend when `/usr/bin/security` is missing.
- [ ] Set / Get / Rm / List / Has / Close implement the same
      semantics as the macOS + Windows backends: ErrNotFound for
      missing entries, sorted names from List, zero-on-Close for any
      cached state, validate names against the same regex.
- [ ] **§4 discipline.** Values move through D-Bus method arguments
      that never appear in process argv or environment. The
      `org.freedesktop.Secret.Service` "Plain" algorithm is sufficient
      for MVP (encryption-at-rest is the keyring daemon's job, not
      ours — same posture as the macOS backend's `security` CLI).
      DH-IETF session crypto deferred to follow-up.
- [ ] **Build gates.** `GOOS=linux go build ./shell/...` clean;
      `GOOS=windows` + `GOOS=darwin` cross-compiles unaffected;
      `make ci` clean.
- [ ] **Live smoke gated by env var** (`AISH_LINUX_BACKEND_LIVE=1`).
      A Linux dev box with `gnome-keyring-daemon` or `kwallet` running
      can exercise the round-trip. CI containers without a session
      bus skip silently (the gate is `errors.Is(err, ErrUnsupported)`).

#### #96 + #98 + #99 Taint system (MVP — command-level sticky-bit)

- [ ] **Parser surface.** `parser.Command.Tainted bool` (new field).
      `parser.Pipeline.Tainted bool` (new field). Default false. Pure
      additive change — existing callers that construct `Command{}`
      literals get zero-value untainted commands without code changes.
- [ ] **Command-substitution source.** When the shell's `RunForCapture`
      executes a sub-pipeline whose argv[0] (after expansion) is
      `secret` and argv[1] is `get`, the resulting captured value is
      marked as a *tainted source* in the shell's per-line
      `TaintedValueRegistry`. The registry stores the literal output
      string → bool tainted. Lookup is exact-match.
- [ ] **Propagation.** After `ExpandLine` (cmdsub) and before
      `Parse`, the shell consults the registry: any token in the
      tokenized pipeline whose value matches a tainted-source literal
      flips the *Command*'s `Tainted` flag. Any tainted Command in a
      Pipeline flips `Pipeline.Tainted` (sticky-bit).
- [ ] **History exclusion.** When `Pipeline.Tainted` is true, the
      history Append writes `[REDACTED:tainted]` as the command line
      instead of the raw `line`. The event still records (kind, cwd,
      timestamp, exit code) — only the command-line column is
      redacted. **Test-asserted via repository scan.**
- [ ] **Direct `secret get NAME | cmd` form.** Even without an
      intervening `$(...)`, a pipeline that *starts* with a
      `secret get` stage is treated as tainted (the secret value
      transits the next stage's stdin). This is the path the live
      smoke covers.
- [ ] **Adversarial test.** Run a tainted pipeline → ripgrep the
      history DB → MUST NOT find the placeholder
      `[REDACTED:test-value-do-not-log]`.

#### #105 Persona-bound secrets

- [ ] **Vault schema reuse.** The `entries[].labels []string` field
      reserved in v0.3-3 carries the active persona name(s).
- [ ] **`secret set` with active persona** tags the entry's `labels`
      with the active persona name (when one is set). Existing entries
      without labels are visible under any persona (backwards-compat).
- [ ] **`secret list` filtering** restricts output to entries that
      either (a) have no labels (always visible) or (b) carry the
      active persona name in `labels`. `secret list --all` overrides
      and shows everything.
- [ ] **`secret get NAME`** is allowed for any entry regardless of
      persona — the *purpose* of persona binding is to scope the
      *visibility surface*, not to gate decryption. (A user who knows
      the name + passphrase can always read it; the gate is `list`.)
- [ ] **Threat model coverage** for the mid-session-switch case is
      documented below.

#### #106 Signed history events — strengthening pass

- [ ] **Strengthening test.** Assert that `secret.get` and
      `identity.use` events written by the shell carry a non-empty
      `Signature` field when a signer is attached.
- [ ] **Persona switch event.** `persona use NAME` also writes a
      `persona.use` history event (new kind, name only).

### Deferred (acknowledged stubs / follow-up issues)

- [ ] **#104 Atomic persona switching — SSH, cloud, kube, git.**
      Cross-cuts SSH config (`~/.ssh/config`), cloud profiles
      (`~/.aws/credentials`, `~/.config/gcloud/`), kube context
      (`~/.kube/config`), git config (`~/.gitconfig`). None of those
      live in the secrets package; they're a separate engine that
      *uses* identity + persona + secrets. The existing
      `identity use NAME --persona <p>` from PR #199 already pairs
      identity + persona atomically. The remaining work (the
      toolchain-level switch) is a follow-up worktree.
      **Filed as follow-up.**

## Alternatives Tables

### A — Taint storage

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Per-line registry of tainted literal strings (MVP, chosen)** | Smallest parser change. The registry lives at the shell layer, not the parser, so the parser stays a pure data-transform. Backwards compatible — empty registry == no-taint == legacy behavior. | Exact-match only. A tainted value that the shell partially edits (e.g. `${val^^}` upper-case) is no longer found by the lookup. Acceptable — no shell-edit forms exist in v0.3-3. | **Chosen.** Cheapest MVP that delivers the "secret value doesn't enter history" promise. |
| Per-token taint bit threaded through the tokenizer | The "right" long-term shape: every `token` struct carries `Tainted bool`. Survives partial edits. | Touches the tokenizer's hot path and every test that constructs tokens. Out of scope for a follow-up wave. | **Deferred** to v0.3-fu-secrets-followup. Documented at the taint module head as TODO(#96-followup). |
| Per-byte/per-rune taint flow | Theoretically pure (every byte of stdin → stdout in a tainted pipeline is tainted). | Requires a process-supervision shim around every external command. Far beyond MVP. | Rejected. |

### B — Linux D-Bus client

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **`github.com/godbus/dbus/v5` (chosen)** | Pure-Go (no CGO), mature, widely used. Already in the local module cache at v5.2.2. Apache-2.0 licensed. | Thicker dep than a raw socket implementation. | **Chosen.** The CGO-free constraint rules out `go-libsecret`. |
| `github.com/keybase/go-keychain/secretservice` | Higher-level Secret Service abstraction. | Pulls keybase's larger keychain abstraction in just to access the Linux backend. License (BSD) is fine but the surface is bigger. | Rejected — the godbus surface is small enough that a direct binding is clearer. |
| Hand-rolled D-Bus over a Unix socket | Zero deps. | Hand-rolling D-Bus auth + message framing is its own follow-up. | Rejected. |
| Shell-out to `secret-tool` (libsecret CLI) | Mirrors the macOS posture (shell out to `security`). | Adds a runtime-binary dependency that is NOT always installed (libsecret CLI is a separate package on most distros). | Rejected — godbus is in the module cache, `secret-tool` is not in the typical desktop image. |

### C — Plugin reverse-call wire-up (deferred, but noted)

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| Append-only `MethodSecretFetch` on `libs/proto/inference`, with the shell as the responder | Minimal protocol change. | Plugin would need to drive a *reverse* JSON-RPC channel — the current dispatcher is one-way (plugin reads stdin, writes stdout). The reverse channel would require either a second pair of pipes or interleaved framing on the existing pair. | **Deferred.** No plugin needs this in v0.3-fu-secrets; deferring keeps the proto contract stable. |
| Env-var fanout (shell sets `AISH_SECRET_<NAME>=<value>` in the plugin's environment at spawn time) | Trivial wire-up. | Violates §4 — secrets in environment variables persist for the lifetime of the plugin process and are visible to anyone who can read `/proc/<pid>/environ`. | Rejected on §4 grounds even if we shipped this issue. |
| Shell-spawned subprocess that pipes the secret to the plugin via a one-shot pipe | The "least worst" if we had to ship today. | Still requires the plugin to know which secrets to ask for at spawn time, which is the design problem this issue exists to solve. | Out of scope. |

## Threat Model (#105 — Persona-bound secrets)

| Threat | Mitigation in this PR | Residual risk |
|---|---|---|
| **Mid-session persona switch leaks the prior persona's secrets** | Persona binding is a *visibility filter* on `secret list`, not a hard gate on `secret get`. Switching persona changes which names `secret list` enumerates; it does NOT zero the vault key or close the vault. A `secret get` to a known name from the prior persona still succeeds. | Documented as intentional — the persona engine is a UX scope, not a security boundary. A user who needs hard isolation creates separate identities (which key separate vaults). |
| **Persona "spillover" from `secret set`** | A `secret set NAME` while persona `work` is active tags the entry with `labels=["work"]`. A subsequent persona switch to `personal` removes it from `secret list` but leaves the value retrievable by name. | Documented in `secret set` help. The user can re-set under a different persona to override the binding. |
| **Stale labels on persona deletion** | A persona deleted from the registry leaves vault entries with that label intact. `secret list` shows nothing from that persona under any active context; `secret list --all` still works. | Documented. A `secret prune-labels` follow-up is filed. |
| **Persona name confusion with secret name namespace** | Persona names use the `[a-z0-9-]` regex; secret names use the `[A-Z][A-Z0-9_]` regex. The character sets are disjoint, so a persona name can never collide with a secret name in any output context. | None. |
| **Filter bypass via `secret list --all`** | The `--all` flag is opt-in and documented. Output is identical to legacy `secret list` (no persona engine awareness). | None — `--all` is the design's intentional escape hatch. |
| **Label leak in vault file** | Labels are stored as plaintext in `vault.json`. A stolen vault file reveals which secrets belonged to which personas. | Documented. Not considered a leak — persona names are not secret. The vault's at-rest cipher protects the secret *values*; the *names* and *labels* have always been plaintext. |

The persona engine is a **UX scope, not a security scope.** This is
the same posture the macOS Finder uses for app-segmented "files"
groupings: it organizes what the user sees, it does not enforce
isolation between groups. The non-overridable security floor for
secrets remains the vault's KDF + AES-GCM.

## Open questions

- **Should `secret list` filter by default once persona is active?**
  Yes — that's the entire point of #105. The escape hatch is `--all`.
- **Does the taint registry survive a `clear`?** Yes — it's
  per-shell-line; the registry is rebuilt per `RunLine`. A new line
  starts with an empty registry.
- **Does sub-shell inherit the registry?** No — sub-shells are
  separate processes. A `$(...)` capture runs inside the parent and
  feeds the captured string back through the registry; nested
  sub-shells are out of scope for v0.3-fu-secrets.
