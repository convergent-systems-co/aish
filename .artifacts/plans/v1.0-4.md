# Plan: v1.0-4 — Windows Secrets

> Wire a **Windows-native secrets backend** behind the `secrets.Backend`
> interface that v0.3-3 (PR #189) reserved. The local Argon2id +
> AES-256-GCM vault remains the cross-platform fallback; this PR adds a
> Windows-only path that lets the user store secrets in **Windows
> Credential Manager** (visible in Control Panel) with values
> additionally wrapped in **DPAPI** (`CryptProtectData`) so a raw dump
> of `vaultsvc` storage still requires the user's logon session key to
> recover plaintext.

**Source:** GOALS.md §"Epic v1.0-4 — Windows Secrets"
**Epic (supervised):** #146
**Tasks (in scope this PR):** #147 (Credential Manager integration —
intentionally CGO-free via `golang.org/x/sys/windows`), #148 (`aish
secret` parity).
**Domain:** `domain:secrets`, `os:windows`
**Risk:** MEDIUM per labels. Windows-only code paths cannot be unit
tested on the macOS dev host — only cross-compiled. UAT happens on the
v1.0-1 UTM Windows VM (TL_WIN_BUILD's deliverable). This plan
intentionally limits live-on-Windows scope to a single binary path
(`backend_windows.go`) so the UAT surface is small.
**Milestone:** v1.0 — Windows Native (120 days)

## Intent / Plan / Risks / Verification / Out-of-scope

### Intent

Ship a `secrets.Backend` interface (the v0.3-3 reservation) and a
Windows implementation that meets four properties:

1. **No CGO.** All Win32 calls go through `golang.org/x/sys/windows`
   or a thin `LazyDLL` binding for the calls x/sys does not expose
   (`CredWriteW` / `CredReadW` / `CredDeleteW` / `CredFree`). The
   issue title says "CGO integration"; the AGENTS.md/Constitution-level
   directive ("no CGO where pure Go reaches") wins. Tracked as a
   documented deviation in this plan (§Risks).
2. **Parity with `aish secret`.** The Backend interface exposes the
   same five operations the local vault implements: `Set / Get / Rm
   / List / Has`. The Windows backend implements all five against
   Credential Manager; values are DPAPI-wrapped before storage and
   DPAPI-unwrapped on retrieval.
3. **Backend selection is compile-time.** This PR does NOT wire the
   Windows backend into the `secret` built-in dispatch path — that's
   a v1.0-2/v1.0-5 follow-up where the built-in chooses between
   `LocalVault` and `WindowsBackend` based on `runtime.GOOS` + an
   identity-profile preference. This PR's deliverable is *that the
   Windows backend exists and cross-compiles and round-trips on
   Windows*, behind the interface the local vault will also satisfy.
4. **Reversible.** A user who runs `aish secret rm NAME` on Windows
   removes the Credential Manager entry. A user who uninstalls aish
   and reinstalls finds their Credential Manager entries intact (DPAPI
   is keyed to the logon session, not the binary).

### Plan

**P1. Define the Backend interface (file: `backend.go`).** Move the
five operations from `Vault` onto an exported interface. Add
`ErrUnsupported`. The `Vault` type keeps its existing method set;
this PR does NOT make `*Vault` implement the interface (that's the
v1.0-5 dispatch refactor — out of scope, see §Out-of-scope).

```go
type Backend interface {
    Set(name string, value []byte) error
    Get(name string) ([]byte, error)
    Rm(name string) error
    List() ([]string, error)
    Has(name string) (bool, error)
    Close() error
}
```

The signatures are *almost* identical to `Vault`'s but differ in two
places on purpose:

- `List` and `Has` return `error` because OS-keychain backends can
  fail at enumeration time (DPAPI unsealed but Cred enumeration
  refused). The local `Vault` will adapt by returning `nil`.
- `Close` is part of the interface; the local vault's `Close()` is
  already on the type.

**P2. Stub backend for non-Windows (file: `backend_other.go`).** Build
tag `//go:build !windows`. `OpenWindowsBackend` returns `nil,
ErrUnsupported` so callers that compile on macOS/Linux get a clean
sentinel rather than a build error.

**P3. Windows backend (file: `backend_windows.go`).** Build tag
`//go:build windows`. Contents:

- `windowsBackend` struct holds the target-name prefix (`"aish:"`) and
  optional entropy bytes.
- `OpenWindowsBackend(prefix string, entropy []byte) (Backend, error)`
  constructor. `prefix` defaults to `"aish:"` if empty.
- `Set(name, value)` — calls `dpapiProtect(value)` to wrap with
  `CryptProtectData(CRYPTPROTECT_UI_FORBIDDEN)`, then calls
  `credWrite("aish:"+name, protectedBlob)` with type
  `CRED_TYPE_GENERIC` and persist `CRED_PERSIST_LOCAL_MACHINE`.
- `Get(name)` — `credRead("aish:"+name)` → `dpapiUnprotect(blob)`.
- `Rm(name)` — `credDelete("aish:"+name)`.
- `List()` — `CredEnumerateW("aish:*", 0, ...)` → strip prefix.
- `Has(name)` — `credRead` and check `ERROR_NOT_FOUND`.
- `Close()` — no-op (no in-memory key state).
- DPAPI calls use `windows.CryptProtectData` / `CryptUnprotectData`
  (already in x/sys).
- Credential Manager calls bind `advapi32.CredWriteW`,
  `CredReadW`, `CredDeleteW`, `CredEnumerateW`, `CredFree` via
  `windows.NewLazySystemDLL("advapi32.dll")` because x/sys does not
  expose them.

**P4. Tests (file: `backend_windows_test.go`).** Build tag
`//go:build windows`. Three behavioral round-trips:

- `set → get → rm` of a single entry returns the original bytes
  exactly.
- `set → list` enumerates the entry by short name (prefix stripped).
- `get` on a non-existent name returns `ErrNotFound`.

A `helper_windows_test.go` (also windows-only) creates a unique
prefix per test run (`"aish-test-" + random-suffix`) so tests are
hermetic across parallel runs and prior failures don't poison the
next run. `t.Cleanup` deletes every entry the test wrote.

**P5. `!windows` test stub (file: `backend_other_test.go`).** Build
tag `//go:build !windows`. Asserts `OpenWindowsBackend` returns
`ErrUnsupported` so the CI macOS path covers the sentinel.

**P6. `go.mod` / `go.sum`.** Promote `golang.org/x/sys` from indirect
to direct (it's already in the graph at v0.44.0 via x/crypto). Single
`go mod tidy` run.

**P7. Cross-compile gate.**
```
GOOS=windows GOARCH=amd64 GOFLAGS=-buildvcs=false go build ./shell/...
GOOS=darwin  GOARCH=arm64 GOFLAGS=-buildvcs=false go build ./shell/...
GOOS=linux   GOARCH=amd64 GOFLAGS=-buildvcs=false go build ./shell/...
make ci  # macOS native
```

### Alternatives Table — DPAPI vs Credential Manager vs both

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **DPAPI only** — store ciphertexts in a JSON file the way `Vault` does, but with `CryptProtectData` instead of AES-GCM | Single file, easy backup, no Credential Manager UI surprise. | Invisible to Windows admin tooling. Loses portability across logon sessions (workgroup) or roaming-profile machines (domain) if not re-encrypted with `CRYPTPROTECT_LOCAL_MACHINE`. Users can't audit "what does aish have?" via Control Panel. | Rejected |
| **Credential Manager only** — `CredWrite` plaintext (Windows-supplied encryption at rest is the LSA secret) | User-visible in Control Panel → Credential Manager → Generic Credentials. Survives reboot. Trivial to enumerate. | Plaintext in process memory on `CredRead` — same as our local vault path, fine — but the bytes-at-rest are protected by an LSA-secret regime that's strong against offline attack but documented as user-session-readable by any process under the same user. We want a defense-in-depth wrap so a same-user process can't `CredEnumerate` + `CredRead` and grab plaintexts without an extra DPAPI unwrap step. | Rejected as solo |
| **Both — Credential Manager as the storage surface, DPAPI as the inner wrap** (RECOMMENDED) | User-visible enumeration of *names*. Values at rest are double-protected (LSA + DPAPI). A malicious same-user process that reads the Credential Manager entry still sees an opaque DPAPI blob, not plaintext. Aligns with the `LocalVault` defense-in-depth pattern (passphrase-derived KEK + per-entry random nonce + GCM auth tag) — the analog here is "logon-session-derived KEK via DPAPI + Credential Manager wrap." | Slightly more code. Two failure modes to surface. | **Chosen** |
| Do nothing — ship v1.0 without Windows secrets, keep `LocalVault` everywhere | Smallest blast radius. | Defeats the "OS Insensitive" pillar — Windows users would be stuck with a JSON vault file that ignores the Windows-native primitive. v1.0 ships Windows-native or it doesn't ship. | Rejected |

### Risks

| Risk | Mitigation |
|---|---|
| Issue #147 title says "CGO integration." This PR is pure-Go (no CGO). | Documented deviation. `golang.org/x/sys/windows` is the modern way to call Win32 from Go — CGO would actually be a regression (cgo-cross is fragile, and the build matrix requires cross-compilation from a Linux CI host). Surface this in the PR body and update GOALS.md if needed. |
| Credential Manager has a max entry size (`CRED_MAX_CREDENTIAL_BLOB_SIZE` = 5 KB on most Windows versions, 2.5 KB on older releases). | Document the limit. `Set` returns a clear error if the *protected* blob exceeds it. Typical API tokens are <1 KB so the practical impact is zero; we surface it rather than bury it. |
| `CredEnumerateW` filter syntax is wildcard-only (`*` glob, not regex). | Use `prefix + "*"` and strip the prefix in user-facing output. Tested in `List`. |
| DPAPI keys are tied to the logon session. A roaming-profile user moving between machines without DPAPI-roaming will see ciphertexts they cannot decrypt. | Document the limitation. v1.0-4 is a single-machine MVP — cross-machine identity sync is v1.5+ territory. |
| Cannot unit-test the Windows backend on the macOS dev host. | (a) Cross-compile gate runs every CI cycle. (b) Build-tagged tests live next to the implementation and run on the v1.0-1 UTM VM (TL_WIN_BUILD owns that gate). (c) `backend_other_test.go` covers the sentinel branch on macOS. |
| `windows.CryptUnprotectData` returns a `*uint16` name pointer the caller is supposed to `LocalFree`. Forgetting that leaks process memory per call. | Wrap unprotect in a helper that always `LocalFree`s the returned name and freshly copies the data into a Go-owned slice. |
| Subtle: `CredRead` returns a heap pointer the caller must free via `CredFree`. Forgetting it leaks the credential blob. | Same pattern — every `CredRead` is wrapped in a function that defers `CredFree`. |
| Test cleanup must delete entries even on test failure or the next run sees ghost entries. | `t.Cleanup` registered immediately after each successful `Set`; per-run unique prefix so a forgotten cleanup is at worst noise in Credential Manager, not a test failure. |

      under Generic Credentials with type "aish" (or generic).

### Out-of-scope

- **Wiring the Windows backend into the `secret` built-in dispatch.**
  The built-in still calls `secrets.OpenVault` unconditionally. The
  selection of `LocalVault` vs `WindowsBackend` happens in a follow-up
  PR (v1.0-2 or v1.0-5) where the identity profile gains a
  `secret_backend = "credential_manager"|"local_vault"` field.
- **Making `*Vault` implement `Backend`.** Cosmetic — would require
  adapting `List() []string` → `List() ([]string, error)`, etc.
  Saving the churn for the follow-up that actually uses the interface.
- **DPAPI machine-scope (`CRYPTPROTECT_LOCAL_MACHINE`).** User-scope
  only this PR. Machine-scope is for service accounts and is a v1.0-5
  concern.
- **Persona-bound Windows secrets.** That's #105 territory.
- **Roaming DPAPI / domain-joined cross-machine recovery.** v1.5+.
