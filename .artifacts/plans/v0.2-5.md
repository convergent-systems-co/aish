## Acceptance Criteria

- [ ] **#72 — Brand schema published from aish.** The wire schema lives
      at `libs/proto/theme/theme.go` and is the source-of-truth a
      Brand-Atoms PR will mirror. Document the on-the-wire v1 schema
      (`schema = "https://theme-atoms.com/schemas/theme-v1.json"`)
      alongside the proto types so the contract is unambiguous. Add a
      `Validate()` method on `proto.Brand` that returns the same errors
      aish would surface at fetch time, so consumers/publishers can
      detect malformed brands before Compile.
- [ ] **#77 — Ten curated bundled brands.** `shell/internal/theme/bundled.go`
      ships ten well-known shell brands as the cold-start floor (works
      with zero network, zero filesystem state). Today: `default`,
      `nord-powerline`, `monokai`. Add: `dracula`, `solarized-dark`,
      `solarized-light`, `gruvbox-dark`, `tokyo-night`, `catppuccin`,
      `one-dark`. Each round-trips through Compile, has a non-empty
      palette and roles, and lists 1+ segments.
- [ ] **#79 — Richer `theme preview <name>`.** Today's preview prints
      two prompt lines. Upgrade to a self-contained sample that shows:
      the brand name + description, all segment renderers populated
      with placeholder data, role swatches (one block per role), and
      the glyph table — without touching the registry's active theme.
      A `--plain` flag suppresses ANSI for capture/snapshot use.
- [ ] **#80 — Persistence preserves sibling sections.** `WriteActiveTheme`
      currently overwrites `config.toml` with only `[theme]`. Sibling
      sections (e.g. `[telemetry]`, future `[prompt]`, `[shell]`) are
      lost. Implement a merge-aware writer: read the existing file,
      replace the `[theme] active = ...` key in place (or append the
      section if missing), preserve all other content (comments,
      blank lines, key order, surrounding sections). Round-trips
      idempotently.
- [ ] **#81 — Performance gates.** Two benchmarks in
      `shell/internal/theme/`:
      `BenchmarkSetActive` — measure `Registry.SetActive(name)` in a
      registry of 100 themes. Hard target: <50ms (50,000,000 ns/op).
      Soft target: <100µs (the design says single-pointer-swap; we
      assert the design holds).
      `BenchmarkColorPrompt` — measure per-character color application
      via `theme.ColorPrompt("x")` over an active compiled theme. Hard
      target: <1µs/op (sub-microsecond, per GOALS).
      Failures error out CI via `go test -bench -benchtime=... -run=^$`
      with explicit ns/op assertions in a regular `t.Run`.

## Alternatives Table (config merge strategy — #80)

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Read-modify-write line-by-line (chosen)** | No new dep. Preserves comments + key order. Matches the existing minimal parser. | Hand-rolled; one more code path to maintain. Cannot handle nested tables (n/a for v0.2-5). | **Chosen** — the existing `ReadActiveTheme` reader is already a line-by-line parser; symmetric writer is a small extension. The full TOML library lands in v0.3-1's RC-file work, not here. |
| Pull `BurntSushi/toml` for round-trip writes | Full TOML compliance; future-proof. | BurntSushi's TOML doesn't preserve comments or key order on write (it re-serialises from the parsed AST). Sibling preservation would still require custom handling. | Rejected — doesn't solve the root problem (comment/order preservation). |
| `pelletier/go-toml/v2` for full round-trip | Preserves comments via the v2 API. | New transitive dep for one writer call. v0.3-1's RC work needs a stronger answer anyway; pulling it in now is premature. | Rejected — defer the dep choice to v0.3-1. |

## Test Strategy

- **proto/theme** — table-driven `Validate()` tests: empty name,
  unknown type, role pointing at missing palette key, malformed hex.
  Each existing test stays green.
- **theme/bundled** — `TestBundledBrands_AllCompile` walks every brand
  through `Compile`, asserts non-empty role colors, non-empty segments,
  prompt char glyph present. New brands enter the bundle here.
- **theme/config (merge writer)** — three round-trip tests:
  1. Empty file → write → file contains only `[theme]`.
  2. File with `[telemetry]` and a comment → write theme → both
     `[telemetry]` and the comment survive; `[theme]` is the last
     section.
  3. File with `[theme]` already set → write a different theme → only
     `active` line changes; surrounding text is byte-stable.
- **theme/perf** — bench + regression tests. The bench targets are
  asserted as plain `t.Run` time-budget assertions so `go test` flags
  them without `-bench`.
- **shell/builtin_theme preview** — capture stdout against the
  enriched preview's known shape (anchored on the name + role labels);
  `--plain` output has zero ESC bytes.

## Out of scope (deferred)

- Live reload (fsnotify watcher on `~/.aish/themes/`) — v0.3-1's RC-file
  work is the right place to wire this once the RC reader is real.
- Theme upload / push to theme-atoms.com — pull-only for v0.2-5; the
  catalog is operator-managed.
- Glyph-fallback detection (NF vs non-NF terminal sniff) — the bundled
  set already covers both ends (monokai/default are no-NF; nord-powerline
  + the new powerline brands are NF-only by declaration). Auto-fallback
  belongs with v0.2-1 ghost-text rendering.
- `theme extends:` base-brand walking — Brand-Atoms emits flattened
  brands; deferred per the comment already in `theme.go`.
