# Plan: v0.3-3 — Identity & Secrets Engine

> A local **vault** (encrypted-at-rest) plus an **identity pointer** that
> together replace ad-hoc `$ANTHROPIC_API_KEY` env-var token handling.
> `aish secret <set|get|list|rm>` accepts values via stdin only, emits
> values via the OS clipboard only, and never echoes a value to stdout,
> stderr, logs, or audit entries. `aish identity <use|list|show|create>`
> selects the active machine identity that the vault is keyed for.
>
> Wave-1 MVP. The 11 backlog tasks split into a **6-task MVP floor** and
> a **5-task deferral set** that ship as stubs or documented follow-ups.
> Deferrals are listed explicitly per `Common.md P3 (Surface, Don't
> Bury)` so future TLs see the gap, not a fake completion.

**Source:** GOALS.md §"Epic v0.3-3 — Identity & Secrets Engine"
**Epic (supervised):** #95
**Tasks (in scope this PR):** #97 (secrets CLI), #100/#101 (subset — local
encrypted vault as the cross-platform fallback; OS-keychain integrations
deferred behind a stub), #102 (identity schema), #103 (identity CLI),
#106 (signed-history hook for `secret get`/`identity use`).
**Tasks deferred this PR:** #96 (taint bit type system in the parser),
#98 (history exclusion via taint), #99 (pipe taint propagation),
#104 (atomic SSH/cloud/kube/git persona switch), #105 (persona-bound
secrets — needs TL_PERSONA's persona engine first).
**Domain:** `domain:secrets`
**Risk:** HIGH per GOALS.md. Zero-tolerance correctness on the crypto
primitives and on the "no plaintext escape" rule. The MVP boundary is
drawn so the high-risk surface lives in **one tiny package**
(`shell/internal/secrets/`) — vault + KDF + identity — with the rest of
the shell only adding a single dispatch branch.
**Milestone:** v0.3 — Real Shell (90 days)

## Acceptance Criteria

### MVP-floor (shipping this PR)

- [ ] **#97 `aish secret set NAME`** — reads value from stdin (TTY:
      no-echo via `golang.org/x/term.ReadPassword`; non-TTY: raw
      stdin). Stores under `~/.aish/vault/vault.json` with the value
      encrypted with AES-256-GCM under a key derived from the user's
      passphrase via Argon2id. First call to the vault initializes it
      and prints the KDF cost parameters once (`P3 — surface the
      cost`). Refuses an empty passphrase and an empty value. Exit
      codes: 0 success, 1 user error, 2 vault/IO error.
- [ ] **#97 `aish secret get NAME`** — decrypts and writes the value
      to the OS clipboard (`pbcopy` / `xclip -selection clipboard` /
      `wl-copy` / PowerShell `Set-Clipboard`). Confirms on stdout with
      the literal string `Value copied to clipboard.` and NEVER echoes
      the value. A missing clipboard binary on a TTY-less host exits
      2 with a clear message — no fallback to stdout. The decrypted
      buffer is zeroed before the function returns.
- [ ] **#97 `aish secret list`** — prints names only, sorted, one per
      line. Names are not secret; values never appear.
- [ ] **#97 `aish secret rm NAME`** — atomic delete + rewrite of the
      vault file. 0600 permissions preserved on rewrite.
- [ ] **Vault file format.** Single JSON file at `~/.aish/vault/vault.json`
      with:
      - `version` (uint),
      - `kdf` block (`algo: "argon2id"`, `time`, `memory_kib`,
        `parallelism`, `salt_b64`),
      - `entries` (map of name → `{nonce_b64, ciphertext_b64,
        created_at, updated_at}`).
      Per-entry nonce is 12 random bytes from `crypto/rand`. Vault
      file permissions: **0600**, enforced on every write via
      `os.OpenFile(..., 0600)` + `os.Chmod(..., 0600)` post-write.
      Vault directory permissions: **0700**.
- [ ] **Identity #102** — `~/.aish/identity.toml` with `[active]
      name`, `gateway_url`, `signer_pubkey_sha256`. Schema pinned by
      Go types in `shell/internal/secrets/identity.go`. If the file is
      missing, the shell prints "no active identity" and `secret`
      commands still work (single-identity MVP keys vault to a fixed
      label `"local"`).
- [ ] **Identity CLI #103** — `aish identity use <name>`,
      `aish identity list`, `aish identity show`, `aish identity
      create <name>`. `create` initializes a profile entry under
      `~/.aish/identities/<name>.toml`. `use` sets the active pointer.
      No remote calls; signer-pubkey hash is set by the user
      explicitly or left empty.
- [ ] **Signed-history hook #106 (subset).** When the v0.1-4 history
      engine is available, every `secret get` and `identity use` writes
      a `kind=secret.get` / `kind=identity.use` history event with the
      NAME only (never the value). The signing primitive is the
      history engine's existing Ed25519 path — this PR plugs in, it
      does not extend the crypto.
- [ ] **No plaintext escape.** Adversarial test: ripgrep over the
      vault file after a write of `"PLAINTEXT_SENTINEL"` MUST NOT find
      the literal `PLAINTEXT_SENTINEL`. Adversarial test: capturing
      stdout+stderr of `secret get` MUST NOT contain the value.
- [ ] **Wrong passphrase fails cleanly.** No timing leak above what
      AES-GCM auth-tag verification gives for free. Error message is
      a single line: `secret: wrong passphrase or vault corrupt`. No
      stack trace, no key material in the panic.
- [ ] **`make ci` passes** from the worktree.
- [ ] **Live smoke:** `aish secret set DEMO_KEY` (paste a fake value),
      `aish secret list` (shows `DEMO_KEY`), `aish secret rm DEMO_KEY`.

### Deferred (acknowledged stubs / follow-up issues)

- [ ] **#96 Taint bit type system in Go.** Requires invasive parser
      changes (every `parser.Token` grows a `Tainted bool`). Cuts
      across pipe operator, command-substitution, parameter expansion.
      Worth its own PR — `domain:secrets` task, but in the **parser**
      not the secrets package. **Stub:** the secrets package exposes
      `secrets.IsTainted(value) bool` returning `false` for now, with
      a `// TODO(#96)` reference so the call sites are reservable.
- [ ] **#98 History event exclusion for tainted values.** Depends on
      #96. **Stub:** secret values never enter history *because they
      never leave the secrets package* — by construction the MVP
      cannot leak via history without #96. Documented in the secrets
      package doc-comment.
- [ ] **#99 Pipe taint propagation.** Depends on #96. **Deferred.**
- [ ] **#100 macOS Keychain integration / #101 Linux libsecret.** The
      **local encrypted vault** ships as the cross-platform floor
      (works on every supported OS, no extra deps). OS-keychain
      backends are an **enhancement layer**, not a substitute, since
      a portable test surface (CI, ephemeral containers) needs the
      local vault anyway. **Stub:** `secrets.Backend` interface in
      the package; `LocalVault` is the only implementation in this
      PR. Keychain/libsecret backends are filed as follow-up tasks
      with the interface as the contract.
- [ ] **#104 Atomic persona switching — SSH, cloud, kube, git.**
      Cross-cuts SSH config, AWS/GCP CLI profiles, kube context, git
      config. None of those touch the secrets package directly;
      they're a separate engine that *uses* identity + vault. Deferred
      to a TL_PERSONA / TL_IDENTITY follow-up.
- [ ] **#105 Persona-bound secrets.** Blocked on TL_PERSONA's persona
      engine (sibling worktree `../aish-persona`). The vault is
      designed so `entries[name].labels []string` can carry a persona
      name in a follow-up without a format break — the field is
      reserved in the JSON schema (`labels` is parsed-but-empty in
      this PR).
- [ ] **#106 full signed-history coverage.** The subset above covers
      `secret get` + `identity use`. Full coverage (every secret
      access from every plugin) needs the JSON-RPC secret-fetch
      handler on the plugin contract, which is TL_PLUGIN's surface.
      Deferred.

## Alternatives Table — KDF

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Argon2id (chosen)** | Memory-hard, OWASP-recommended for password storage 2023+, stdlib via `golang.org/x/crypto/argon2`. Tunable. Resistant to both GPU and ASIC brute-force. | Memory cost makes very low-RAM hosts feel the unlock latency. We accept this — the alternative is a weaker KDF. | **Chosen.** Cost parameters: `time=3 iterations`, `memory=64 MiB`, `parallelism=4`, `keyLen=32` bytes. Printed on first vault init. |
| scrypt | Also memory-hard, stdlib via `golang.org/x/crypto/scrypt`. Older and battle-tested. | Memory/CPU tradeoff knobs are coupled; harder to tune defensively without forking. Not Argon2's PHC winner. | Rejected. |
| PBKDF2-SHA256 | Stdlib `golang.org/x/crypto/pbkdf2`, ubiquitous. | Not memory-hard. GPU-friendly. OWASP 2023 recommends Argon2id/scrypt over PBKDF2 for new designs. | Rejected. |

## Alternatives Table — Vault file format

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Single JSON with encrypted blobs (chosen)** | One file → one `fsync`, one chmod. Trivially backed up. Atomic rewrite via tempfile+rename. No new dep. Diffable structure (names visible, values opaque). | Whole-file rewrite on every change. Acceptable: vaults are small (KB range). | **Chosen** |
| SQLite with encrypted columns | Familiar shape; the cache already uses sqlite. Better for very large vaults. | New dep on the secrets-critical path; `modernc.org/sqlite` already vendored but pulling it into the security boundary inflates the audit surface. Sharing a journal/WAL file complicates the 0600/atomic-rename story. | Rejected. |
| Sharded files (one per secret) | Per-entry atomic write; easier rotation. | Filesystem listing reveals names by directory listing (same as JSON, but with more `stat` syscalls). More files → more chmod surface to get wrong. | Rejected. |

## Alternatives Table — Identity scope

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Single identity per machine (MVP, chosen)** | Trivial schema. Matches the v0.3 "real shell" floor. The vault is keyed for the active identity; switching identities means a new vault file (or a future labeled-section format). | A user with `work` + `personal` identities has to re-`secret set` for each. Acceptable for MVP. | **Chosen** |
| Multi-identity profiles with a profile-scoped vault | Maps to the GOALS.md vision (work/personal/SOC2). | Format change to vault to add a profile dimension; identity switching becomes a vault-level operation, raising the test matrix sharply. | **Deferred** — vault JSON schema reserves room for a future `profile` key on each entry without a breaking change. |

## Threat Model

| Threat | Mitigation in this PR | Residual risk |
|---|---|---|
| **Stolen vault file alone** (laptop theft, backup leak) | Argon2id + AES-256-GCM. Adversary needs the passphrase to read any entry. KDF cost printed on init so the user can choose higher if they're a high-value target. | Brute-force is bounded by KDF cost. A weak passphrase is still bypassable — documented in the `secret set` help text. |
| **Vault-in-memory dump while unlocked** | Cleartext values zeroed after each `secret get` (the buffer is wiped before return). The session-scoped key cache holds the KDF-derived 32-byte key, **not** secret plaintexts — values are decrypted on demand and zeroed on return. | A process-memory snapshot taken *during* a `secret get` call can capture plaintext. Mitigating this fully needs `mlock`/`madvise(DONTDUMP)`, which is **out of scope this PR** (filed as follow-up). The session-key cache is also memory-resident — same caveat. |
| **Swap-leak of cleartext or key** | Same as above. `mlock` deferred. Documented in the package doc-comment so the limitation is visible to anyone integrating. | Real, but bounded — swap is encrypted by default on macOS and most modern Linux distros. |
| **Passphrase brute-force (offline)** | Argon2id at 64 MiB / 3 iterations / parallelism=4 puts a single guess in the 250–500 ms range on modern hardware. At 8-char passphrase entropy that's still cracking territory; at 16-char it's intractable. | Documented in `secret set` help. |
| **Plaintext leaking via stdout/stderr/logs** | `secret get` writes to clipboard only. Tests assert no plaintext appears in the captured stdout/stderr of any built-in. Error messages reference `[REDACTED:secret-value]` if a value ever has to be named in a log line (it never does in this PR). | Plugin-side leaks are out of scope here — TL_PLUGIN owns that boundary. |
| **Plaintext leaking via history events** | `secret get` writes a history event with `kind=secret.get` + `name=NAME`. No value field. Test-asserted. | Same caveat re: full #96 taint coverage. |
| **Timing-leak on wrong passphrase** | AES-GCM auth-tag verification + Argon2id (constant-time) gives the floor. No early-return based on partial decryption. | Argon2id verification is constant-time; AES-GCM tag check is constant-time. The OS-level fs/IO timing differences are not in this PR's threat model. |
| **Vault corruption** | Atomic rewrite via tempfile + `os.Rename`. A truncated `vault.json` fails to parse → user sees the error, can restore from backup. No partial-decrypt path. | A user with no backup loses their vault. Documented. Backup guidance in `aish secret --help`. |
| **Prompt-injection via secret names** | Names are never re-interpreted as code. Stored as JSON keys, displayed by `list` with no shell quoting. Reserved-character set: any name not matching `^[A-Z][A-Z0-9_]{0,63}$` is rejected at `set`. | Bounded by the regex. |
| **Empty-passphrase acceptance** | Rejected at both `set` (first init) AND `get` (rejected before KDF) so an attacker cannot create a "vault" with no password and then claim the user had one. | None. |
| **Wrong-permission vault file** | Vault file is chmod'd to 0600 on every write. If the existing file has any other-readable bit set on read, the shell prints a warning to stderr. | Filesystems that don't honour POSIX modes (Windows NTFS without WSL) — out of scope this PR (TL_WIN32). |

## Approach

1. **`secrets` package primitives.**
   1. `aead.go` — `Seal(key, nonce, plaintext)` / `Open(key, nonce,
      ciphertext)` thin wrappers around `crypto/aes` + `cipher/gcm`,
      with `Zero(buf)` helper. Deterministic test vectors.
   2. `kdf.go` — `Derive(passphrase, salt, params) ([32]byte, error)`
      using `argon2.IDKey`. `DefaultParams()` returns `{Time: 3,
      Memory: 64*1024, Parallelism: 4, KeyLen: 32}`. `Cost()` prints
      a single-line description for the init message.
   3. `passphrase.go` — `Read(prompt string) ([]byte, error)`. If
      `term.IsTerminal(0)` then `term.ReadPassword(0)`; else read a
      single line from stdin and strip the trailing newline. Empty
      input returns an error.
   4. `clipboard.go` — `Copy(value []byte) error`. Detect `pbcopy`,
      `xclip`, `wl-copy`, `Set-Clipboard` on PATH; write via `exec.Cmd`
      Stdin pipe. NEVER touch stdout/stderr of the parent process.
2. **Vault file format.**
   1. `vault.go` — `Vault` struct backed by `~/.aish/vault/vault.json`.
      `OpenVault(home, passphrase)` reads + decrypts on demand (the
      vault holds only the derived key in memory, not plaintexts).
      `Save()` writes atomically via tempfile + rename, chmod 0600.
   2. `Set(name, value)` → encrypt under per-entry nonce + Save.
   3. `Get(name)` → Open the entry's ciphertext, returns the
      plaintext; caller is responsible for zeroing.
   4. `List()` → returns sorted names.
   5. `Rm(name)` → delete + Save.
3. **Identity.**
   1. `identity.go` — `LoadActive(home)` reads
      `~/.aish/identity.toml`. `SetActive(home, name)` rewrites it.
      `CreateProfile(home, name)` writes
      `~/.aish/identities/<name>.toml`.
4. **Built-ins.**
   1. `builtin_secret.go` — dispatch table `set | get | list | rm |
      help`. Each sub-command reads passphrase once (cached on the
      `Shell` for the session — see step 5).
   2. `builtin_identity.go` — dispatch table `use | list | show |
      create | help`.
   3. `shell.go` — two added branches in `dispatch`, two entries in
      `ResolveTier`.
5. **Session key cache.** The first `secret <subcommand>` per session
   prompts for the passphrase and caches the derived 32-byte key on
   the `Shell` struct. Subsequent calls reuse the key without
   re-prompting. The cache is in-memory only; process exit zeroes
   it. (Optional `secret lock` command zeroes early — defer if time
   short.)
6. **History hook.** When `s.history != nil`, every `secret get`
   and `identity use` appends a `Kind: "secret.get"` /
   `Kind: "identity.use"` event with the NAME only.

## Testing strategy

- **Crypto round-trip.** Deterministic Seal → Open with fixed key +
  nonce equals plaintext. Tampered ciphertext byte → auth-tag error.
- **KDF determinism.** Same passphrase + salt → same key (no
  nondeterminism). Different salt → different key.
- **No plaintext on disk.** Set `"PLAINTEXT_SENTINEL"`, read the raw
  vault file with `os.ReadFile`, assert `bytes.Contains(file,
  []byte("PLAINTEXT_SENTINEL"))` is **false**.
- **No plaintext on stdout/stderr.** Capture both during `secret get`,
  assert the value does not appear.
- **0600 permissions.** Stat the vault file post-write, assert
  `info.Mode().Perm() == 0o600`.
- **Wrong passphrase.** OpenVault with the wrong passphrase →
  expected single-line error, no panic, no value disclosure.
- **Empty passphrase rejected.** At set-init and at get-time both.
- **No real-looking secret literals.** Every test uses
  `"test-fake-value-<n>"` style placeholders; no hex strings that
  look like keys, no `sk-…` / `xoxb-…` prefixes.
- **Built-in dispatch.** `ResolveTier("secret")` /
  `ResolveTier("identity")` both report `term.TierBuiltin`.
- **Live smoke.** Bash script under `shell/internal/shell/` runs the
  three-step smoke test against a tempdir-rooted `$HOME`.

## Risk assessment

- **Crypto-on-wrong-defaults.** Mitigated by stdlib-only / x/crypto-only
  primitives and by deterministic test vectors that pin the AEAD
  contract.
- **0600 surface drift.** Centralized in `vault.atomicWrite()` so
  there's exactly one chmod call.
- **Clipboard binary missing on minimal Linux containers.** Tests
  skip when no clipboard binary is on PATH; the `get` built-in fails
  loudly rather than falling back to stdout. Documented.
- **history hook coupling.** Behind a small adapter interface; if
  the history package doesn't yet expose the needed `Append`
  method, the hook degrades to a no-op (logged once, never blocks
  the secret op).

## Dependencies

- New direct dep: `golang.org/x/crypto/argon2`. Already in the Go
  ecosystem; `go mod tidy` resolves.
- No new runtime binaries required beyond what the OS provides
  (`pbcopy` on macOS is built-in; on Linux the user must have
  `xclip` or `wl-clipboard` if they want clipboard support).

## Backward compatibility

- Pure-additive surface — new package, new built-ins, no existing
  behavior changes. Pre-existing `$ANTHROPIC_API_KEY` env-var still
  works for the cloud plugin; this PR does not yet rewire the cloud
  plugin to ask the vault (TL_PLUGIN's territory).
- Vault file format carries a `version: 1` field; future
  multi-identity profiles add a `version: 2` upgrade path.

## Out of scope

- **Parser-level taint propagation (#96, #98, #99).** Belongs in the
  parser; orthogonal to the vault.
- **OS keychain backends (#100, #101).** Reserved by the
  `secrets.Backend` interface so a follow-up PR can implement them
  without touching callers.
- **Atomic persona switching (#104).** Cross-cuts SSH, cloud CLIs,
  kube, git — a separate engine that *uses* identity + vault.
- **Persona-bound secrets (#105).** Blocked on TL_PERSONA.
- **Full plugin-side secret-fetch JSON-RPC (#106 full scope).**
  TL_PLUGIN's territory.
- **`mlock` / swap-leak hardening.** Documented limitation, follow-up.
- **Vault export/import.** Follow-up.
- **Passphrase change / KDF re-derivation.** Follow-up — needs a
  `secret rekey` command that re-encrypts every entry.
