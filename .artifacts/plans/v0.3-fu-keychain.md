# Plan: v0.3-3 follow-ups — macOS Keychain + Linux Secret Service

> Add the two remaining OS-native secrets backends behind the `Backend`
> interface that v0.3-3 (PR #189) introduced and v1.0-4 (PR #192) first
> wired (Windows DPAPI + Credential Manager). This PR adds the **macOS
> Keychain** and **Linux freedesktop Secret Service** implementations,
> closes #100 and #101, and leaves the local Argon2id + AES-256-GCM
> vault as the cross-platform fallback.

**Source:** GOALS.md §"Epic v0.3-3 follow-ups" (deferred from v0.3-3)
**Issues (in scope this PR):** #100 (macOS Keychain), #101 (Linux libsecret / Secret Service)
**Sibling PRs:** TL_BUILTINS (`feat/v0.3-fu-builtins`), TL_PERSONA_FU (`feat/v0.3-fu-persona`)
**Domain:** `domain:secrets`, `os:macos`, `os:linux`
**Risk:** MEDIUM. macOS code is testable on the dev host; Linux requires either an integration host with a running Secret Service daemon, or cross-compile-only gating.
**Milestone:** v0.3 — Real Shell (90 days)

## Intent / Plan / Risks / Verification / Out-of-scope

### Intent

Two new `Backend` implementations:

1. **macOS Keychain** (`backend_darwin.go`, `//go:build darwin`). Pure-Go;
   shells out to `/usr/bin/security` (`add-generic-password`,
   `find-generic-password`, `delete-generic-password`,
   `dump-keychain`). Same interface as Windows; entries land in the
   user's login keychain visible in Keychain Access.app.
2. **Linux freedesktop Secret Service** (`backend_linux.go`,
   `//go:build linux`). Pure-Go via `github.com/godbus/dbus/v5`
   speaking the Secret Service D-Bus protocol directly. On a session
   without a Secret Service daemon (most server distros), `OpenLinuxBackend`
   returns `ErrUnsupported` so the dispatch layer can fall through
   to LocalVault cleanly — no panic, no half-state.
3. **`backend_other.go`** extends its build tag from `!windows` to
   `!(windows||darwin||linux)` so the sentinel still covers genuinely
   unsupported platforms (BSDs, Plan 9, JS/WASM).

The Backend interface itself does **not** change in this PR. No edits
to `backend.go`, the vault, KDF, AEAD, or any consumer.

### Plan

**P1. Extend `!windows` stub to `!(windows||darwin||linux)`.**
Edit `backend_other.go` build tag. Same body, same `OpenWindowsBackend`
sentinel. Add a sibling `OpenDarwinBackend` and `OpenLinuxBackend`
sentinel in the same file (so callers can reference all three symbols
on any platform without a build error).

**P2. macOS backend (file: `backend_darwin.go`).** Build tag
`//go:build darwin`.

Strategy: shell out to `/usr/bin/security`. The binary is on every
macOS install (part of the OS, not an SDK). No CGO. No third-party
deps. Slight per-call latency (~10ms sub-process spawn) is acceptable
for an interactive shell built-in.

- `OpenDarwinBackend(service string, account string) (Backend, error)`
  constructor. `service` defaults to `"aish"` (the `-s` flag to
  `security`); each entry's `name` becomes the `-a` (account) field
  with a per-backend `account` prefix (defaults to `""`).
- `Set(name, value)` — `security add-generic-password -U -s aish -a name -w value`.
  The `-U` flag is "update if exists" — overwrite semantics.
- `Get(name)` — `security find-generic-password -s aish -a name -w`.
  The `-w` flag prints just the password to stdout. We pipe stdin
  closed and read stdout exactly.
- `Rm(name)` — `security delete-generic-password -s aish -a name`.
  Exit code 44 (`errSecItemNotFound`) maps to `ErrNotFound`.
- `List()` — `security dump-keychain` is expensive and dumps every
  service. Instead: keep a hint by enumerating with `find-generic-password`
  in a loop is also wrong (no "list by service" command). Two options:
  - **Chosen:** track a small JSON file `~/.aish/keychain-index.json`
    of names this backend has written. Names are non-secret. Vault file
    permissions (0600) apply. The index is rebuilt on Set, pruned on
    Rm. List() reads it and verifies each name is still present via
    `Has`.
  - Rejected: `security dump-keychain -d` requires unlock and prompts
    for every item — bad UX.
- `Has(name)` — `security find-generic-password -s aish -a name`
  (no `-w`). Exit 0 → exists; exit 44 → not found.
- `Close()` — Zero index buffer; mark closed.
- Tests gated by `AISH_KEYCHAIN_INTEGRATION=1` env var (popup prompts
  if the keychain is locked). Default `go test` skips them cleanly.
- A unit-level test for the index file rebuild / prune logic runs
  unconditionally (no `security` calls).

**P3. Linux backend (file: `backend_linux.go`).** Build tag
`//go:build linux`.

Strategy: speak D-Bus directly via `github.com/godbus/dbus/v5`. The
Secret Service protocol is small: `org.freedesktop.secrets.Service`
(`OpenSession`, `CreateCollection`, `SearchItems`), and per-item
`SetSecret`, `GetSecret`, `Delete`. Pure Go. No CGO. The package is
~150 KB compressed. No libsecret dependency.

- `OpenLinuxBackend(service string, schema string) (Backend, error)`
  constructor. `service` defaults to `"aish"`. Connects to the session
  bus, opens a `plain` session (no encryption between us and the
  daemon — fine, IPC is local-only Unix socket). If `dbus.SessionBus()`
  fails (no `DBUS_SESSION_BUS_ADDRESS` env or no broker), returns
  `ErrUnsupported`. If the Secret Service interface isn't on the bus
  (e.g., `gnome-keyring-daemon`/`KeePassXC`/`kwallet` not running),
  also returns `ErrUnsupported`.
- Uses the default collection (`/org/freedesktop/secrets/aliases/default`).
  Items tagged with attributes `{"service": "aish", "name": NAME}` so
  enumeration via `SearchItems` is exact.
- `Set / Get / Rm / Has / List` map to `CreateItem` (with replace=true),
  `Item.GetSecret`, `Item.Delete`, `SearchItems` for membership, and
  `SearchItems` with just `service=aish` for the full list.
- `Close()` — closes the D-Bus connection, zeroes any session-key
  state, marks closed.
- Tests gated by `AISH_SECRET_SERVICE_INTEGRATION=1`. Default `go test`
  exercises only the constructor's "no daemon" fall-through.

**P4. Cross-platform compile guard.** Ensure
`go.mod` picks up `github.com/godbus/dbus/v5`. The library is pure-Go
and cross-compiles fine — we depend on it on darwin/windows builds too
(even though it won't be exercised) to keep the module graph uniform.

**P5. `go.mod` / `go.sum`.** Single `go mod tidy` after adding the
import.

**P6. Cross-compile gate.**
```
GOOS=windows GOARCH=amd64 GOFLAGS=-buildvcs=false go build ./shell/...
GOOS=darwin  GOARCH=arm64 GOFLAGS=-buildvcs=false go build ./shell/...
GOOS=linux   GOARCH=amd64 GOFLAGS=-buildvcs=false go build ./shell/...
make ci  # macOS native (full test suite incl. macOS backend unit tests)
```

### Alternatives Table — macOS strategy

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Shell out to `/usr/bin/security`** (RECOMMENDED) | Zero deps. Zero CGO. Binary present on every macOS install since 10.x. Behavior auditable (same commands a sysadmin would type). | ~10ms per call sub-process latency. stdout / stderr have to be parsed carefully so a `security` binary swap can't smuggle bytes back into the process. | **Chosen** |
| Pure-Go `github.com/keybase/go-keychain` (MIT) | One library call, no spawn. Battle-tested in keybase/client. | Uses CGO to link `Security.framework`. Adds ~1 MB to binary. Requires Xcode CLT in dev environments without it. Defeats the "no CGO" property the Windows backend establishes. | Rejected |
| Direct CGO bindings to `SecKeychainAddGenericPassword` et al | Maximum control, no third-party. | Same CGO objections. More code to maintain. | Rejected |
| Defer macOS support entirely, ship Linux only this PR | Smallest blast radius. | #100 stays open; macOS users (the dev host) lose feature parity with Windows. Half-deliverable. | Rejected |

### Alternatives Table — Linux strategy

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **D-Bus Secret Service via `github.com/godbus/dbus/v5`** (RECOMMENDED) | Pure-Go. The de-facto Go D-Bus client. Speaks Secret Service protocol directly — no wrapper layer to debug. Cross-compiles to non-Linux as a no-op import. | Have to hand-roll the Secret Service protocol calls (CreateItem, GetSecret, etc.). ~200 LoC of D-Bus marshaling. | **Chosen** |
| CGO bindings to `libsecret` | One-call API. Battle-tested. | CGO. Requires libsecret-1-dev at build time on every Linux CI host. Defeats the "no CGO" property. Hairier cross-compile. | Rejected |
| `github.com/keybase/go-keychain/secretservice` | Pure-Go wrapper for Secret Service. MIT. Same maintainer as the macOS keychain library. | Adds a transitive dep on the keybase project. Wrapper is thin — marginal time savings over godbus direct. Less control over error mapping. | Rejected |
| Shell out to `secret-tool` (libsecret CLI) | Symmetric with macOS strategy. | `secret-tool` is NOT installed by default on most distros — adds a runtime prerequisite. Fragile. | Rejected |
| Defer Linux support, ship macOS only this PR | Smallest blast radius. | #101 stays open; Linux is the primary aish target by user count. Half-deliverable. | Rejected |

### Risks

| Risk | Mitigation |
|---|---|
| macOS Keychain prompts the user on Set / Get if the login keychain is locked. | The `security` CLI inherits the keychain's unlock state. Document the limitation. Tests guarded by `AISH_KEYCHAIN_INTEGRATION=1` so CI never blocks on a prompt. |
| `security` binary swap (attacker writes a shim at `/usr/bin/security`). | Path is hardcoded to `/usr/bin/security` (absolute), bypassing `$PATH`. macOS SIP protects `/usr/bin/security` from non-root writes. Defense in depth: a tampered system binary is well outside our threat model. |
| `security` stdout includes a trailing newline. | Strip exactly one trailing `\n` if present; do NOT trim arbitrary whitespace (a value ending in space is valid). |
| `security` writes the password to stderr instead of stdout with `-w` on older macOS. | Pinned behavior since macOS 10.10. Document the minimum macOS version (Big Sur / 11.0+) in the package doc. |
| Linux: no Secret Service daemon running (servers, headless installs). | `OpenLinuxBackend` returns `ErrUnsupported`. Dispatch falls through to LocalVault. Tested with `unset DBUS_SESSION_BUS_ADDRESS` simulation. |
| Linux: multiple Secret Service providers (gnome-keyring + KeePassXC + kwallet) — which wins? | Whichever holds the `org.freedesktop.secrets` bus name. We don't fight this — first-mover wins per freedesktop spec. Document the behavior. |
| Linux: Secret Service items can outlive the aish process if the user removes the local vault. | Document the cleanup path: `aish secret rm NAME` deletes via D-Bus. `aish secret list` enumerates only entries with our `service=aish` attribute, so no foreign items show up. |
| macOS keychain-index file is non-authoritative — keychain entries deleted out-of-band drift from the index. | List() does a `Has` cross-check per name and silently prunes stale index entries. Documented behavior; covered by unit test. |
| `godbus/dbus/v5` adds a non-stdlib dep. | Pure-Go, ~150 KB compressed, MIT, well-maintained (active commits in last 60d), used by snapd / juju / cilium. Acceptable. |
| Tests for the index-file pruning logic could accidentally bake a secret value. | Use placeholder `"test-fake-value"` literals per `Common.md §4`. No real-looking tokens. |
| Linux test path requires `dbus-launch` + a running Secret Service daemon. | Hard skip via env var (`AISH_SECRET_SERVICE_INTEGRATION=1`). Default `go test` exercises only the constructor's negative path. |
| macOS test path requires user keychain to be unlocked (avoid GUI prompts in CI). | Same gating: `AISH_KEYCHAIN_INTEGRATION=1`. The unit-level index-file tests need no keychain. |

### Out-of-scope

- **Wiring the new backends into the `secret` built-in dispatch.** The
  built-in still calls `secrets.OpenVault` unconditionally. Selection
  of LocalVault / WindowsBackend / DarwinBackend / LinuxBackend
  happens in TL_BUILTINS' sibling PR or a v1.0-5 follow-up.
- **Making `*Vault` implement `Backend`.** Still cosmetic — `List()
  []string` vs `List() ([]string, error)` mismatch. Same deferral as
  v1.0-4.
- **Persona-bound macOS / Linux secrets.** That's #105 territory —
  TL_PERSONA_FU's sibling PR.
- **`SecureEnclave` / `kwallet` direct integration.** macOS Secure
  Enclave is per-app and CGO-only; kwallet has its own D-Bus interface
  distinct from Secret Service. Out of scope for v0.3 follow-ups.
- **Roaming keychains / iCloud Keychain.** Single-machine MVP.
- **Linux: prompting for a master password to unlock the keyring.**
  We don't drive `gnome-keyring-daemon`'s prompt flow — if the keyring
  is locked, the D-Bus call returns an error, we surface it, the user
  unlocks via their DE.
