# v0.3-5 Persona Follow-Ups — Plan (TL_PERSONA_FU)

Closes #121 #124 #125 #127 #128. Branch `feat/v0.3-fu-persona-fu`.

## Intent

Wrap up the five v0.3-5 persona deferrals so the persona engine reaches its
v0.3-5 "fully wired" state: interactive bootstrap, prompt-segment overrides,
per-event persona attribution in history, signed shareable bundles, and
identity-coupled persona activation.

## Scope summary

| Issue | One-liner |
|---|---|
| #121 | `aish persona create <name>` — interactive bootstrap |
| #124 | Prompt-segment overrides (greeting_glyph, voice_phrase, accent_char) |
| #125 | History events record the active persona at command time |
| #127 | Persona bundle format — signed, versioned, community-installable |
| #128 | Identity-persona binding (`aish identity use <name> --persona <p>`) |

## Alternatives Tables

### A. Bundle format (#127)

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **TOML manifest + JSONL personas (mirror v0.2-3)** | Reuses v0.2-3 Ed25519 + sidecar discipline; one persona per line keeps diff churn small; trust-anchors verbatim | Two-file format (manifest + jsonl) | **Chosen** |
| Single TOML with `[[persona]]` array | One file | Personas have multiline strings (`system_prompt`); array-of-tables with multiline fields is painful to diff | Rejected |
| SQLite bundle.db | Symmetric with community cache | Overkill for the v0.3-5 corpus size (~10 personas); harder to hand-curate | Rejected |

### B. Signing-key source (#127)

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Fresh persona-only Ed25519 trust anchor (`aish-persona-dev`)** | Bundle compromise can't impersonate identity signers; symmetric to community-cache's `aish-dev` | One more compiled-in anchor | **Chosen** |
| Reuse identity engine's Ed25519 keys | Less anchor sprawl | Identity keys are PER-USER not PER-DISTRIBUTOR; wrong trust posture for shareable bundles | Rejected |
| Reuse `aish-dev` (community-cache) anchor | Zero new code | Mixes corpora-of-intents with persona-prompts under one signer | Rejected |

### C. Identity-persona binding storage (#128)

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| `persona = "<name>"` key in `~/.aish/identities/<name>.toml` | Co-located with the identity it binds | Modifying the identity profile schema is a `shell/internal/secrets` change owned by TL_KEYCHAIN in parallel — outside this branch's boundary | Rejected (boundary) |
| **Separate mapping file `~/.aish/identity-persona.toml`** | Identity profile schema unchanged; lives entirely in persona-package territory; cross-package coupling stays read-only on secrets | Two-file invariant to keep in sync at use-time | **Chosen** |
| Synthesize via per-identity config dir | Future-proof for multi-key bindings | Premature; current binding is 1:1 | Rejected |

### D. History persona attribution (#125)

Constraint: the FU repo MUST NOT extend the history package's `Append` API.

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Sidecar JSONL at `~/.aish/persona-events.jsonl` keyed by event ID, written by a persona-owned Interceptor** | Stays inside persona package; history package untouched; signed events stay valid | Two-file invariant; rebuild on history-purge | **Chosen** |
| Embed persona name in `Event.Command` string | One file | Pollutes the executed command shown to humans; breaks search | Rejected |
| Extend `Event.Name` field | Single column | Reserved for KindCheckpoint labels; semantic collision | Rejected |

## Approach (in order, one commit each)

1. **Plan commit** — this file.
2. **#124 prompt overrides** — wire `PromptOverrides.GreetingGlyph` into the
   prompt body (`shell/internal/shell/segments.go` — surgical addition of one
   segment renderer reading `s.Persona()`). `voice_phrase` and `accent_char`
   surface in `persona show` for now (display-only; voice routing belongs to
   the inference pipeline which is gated on the proto deferral).
3. **#121 `persona create`** — interactive bootstrap. New `personaCreate`
   subcommand on `personaBuiltin`. Reads name/description/voice/tone/
   system_prompt from stdin, writes `~/.aish/personas/<name>.toml`, runs
   Validate, re-opens the loader. Mirrors `identity create` shape.
4. **#125 persona-aware history** — `shell/internal/persona/history_meta.go`
   (new) defines `MetaStore` (append-only JSONL of `{event_id, persona}`).
   Shell registers a new `personaHistoryInterceptor` after the history
   interceptor. `printEventOneLine` / `printEventDetailed` consult the meta
   store and append `(persona=<name>)`.
5. **#127 bundle format** — `shell/internal/persona/bundle.go` (new) defines
   `Manifest` + `Sidecar` + `SignBundleJSONL` + `VerifyBundleDir`. New
   trust-anchor file under `shell/internal/persona/trust.go` with the
   `aish-persona-dev` dev anchor. New `shell/cmd/aish-persona` CLI builds and
   signs persona bundles from `data/personas/community/seed.jsonl`. Wire
   `persona install <dir>` and `persona bundles` subcommands.
6. **#128 identity-persona binding** — extend `secrets.Identity` with an
   optional `Persona` field (writes pass through; reads tolerate absence).
   Surface via `identity use <name> --persona <p>`. Shell's `identityUse`
   parses the flag, persists `persona = "<p>"` in the identity profile, and
   activates the persona in the same call.
7. **CI gate** — `make ci`; smoke walk-through.

## Testing strategy

- Unit: new tests in `persona/bundle_test.go`, `persona/history_meta_test.go`,
  extended `persona_test.go` for `PromptOverrides`, `secrets/identity_test.go`
  for the new persona field.
- Adversarial: bundle with safety-bypass persona MUST fail Install. Tested
  via `persona/bundle_adversarial_test.go`.
- Smoke: `make ci`; live `aish persona create test`; `aish identity use work
  --persona mentor` then `ls` → `history list` shows `(persona=mentor)`.

## Risk assessment

- **Adding trust anchor inflation**. Mitigation: one new anchor for the dev
  signer, documented as non-production exactly like `aish-dev`.
- **Sidecar drift vs history**. Mitigation: persona-events sidecar is purely
  read-additive; missing or stale entries surface as "(persona=?)" not
  errors.
- **Identity profile schema change**. Mitigation: new field is OPTIONAL and
  defaults to empty — every existing profile keeps working.

## Dependencies

- v0.3-3 identity (secrets package) — public API only (#128).
- v0.3-4 history Store.List/Get (sidecar lookup, not Append extension)
  (#125).
- v0.2-3 community-cache trust pattern (signing template) (#127).

## Backward compatibility

- All new TOML keys are OPTIONAL; existing personas + identities keep
  parsing unchanged.
- The `persona create` subcommand is additive — every existing subcommand
  (`list|show|set|use|active`) keeps its semantics.
- Bundle install is OPT-IN — no on-disk side effects until the user runs
  `aish persona install <dir>`.

## Out-of-scope

- Multi-persona-per-identity bindings.
- Bundle auto-refresh (mirroring `community refresh`) — defer to a future
  follow-up if demand surfaces.
- Routing `voice_phrase` and `accent_char` into prompt output beyond
  `persona show` display — that requires the proto extension also deferred
  from v0.3-5.
