# Plan: v0.2-2 — PTY Support

> Wave 2 of v0.2 layer polish. Interactive programs (`vim`, `less`,
> `top`, `htop`, `ssh`, `az login`) need a real PTY to render and read
> input correctly. Today aish wires children's stdin/stdout/stderr to
> `os.Pipe`s in `exec.Run`, so any program that checks `isatty(0|1)`
> downgrades to non-interactive mode (or breaks outright on raw-mode
> ioctls). This epic gives interactive children a controlling TTY and
> forwards window-size + the foreground-relevant signals.

**Source:** GOALS.md §"Epic v0.2-2 — PTY Support" + §"Signal Handling and Job Control"
**Epic (supervised):** #52
**Tasks (in-scope this PR):** #53, #54, #55, #56, #57
**Domain:** `domain:shell-runtime`
**Risk:** MEDIUM (per GOALS.md — cross-platform PTY behaviour, signal
fidelity, terminal-state restoration on abnormal child exit)
**Milestone:** v0.2 — Layer Polish

## Acceptance Criteria

- [ ] **#53** `github.com/creack/pty` vendored into `shell/go.mod`. BSD-3
      licensed; pure-Go on Unix; widely used. Pinned to a stable tag.
      The dependency is gated to Unix (`//go:build !windows`) in
      `pty_unix.go`; Windows uses a separate file that returns
      `errPTYUnsupported` until ConPTY support lands in a follow-up.
- [ ] **#54** `exec.RunPTY` — new pipeline-of-one entrypoint that
      allocates a PTY, starts the child with its controlling TTY pointed
      at the slave side, and copies bytes in both directions between the
      caller's `*os.File` stdin/stdout and the master fd. Returns the
      child's exit code with the same contract as `exec.Run`. Pipelines
      with `len(p.Commands) != 1` are rejected with `errPTYPipeline` —
      PTY-piping a `cmd1 | cmd2` chain is out of scope; the existing
      stdio path handles those.
- [ ] **#55** Signal forwarding while the PTY child runs:
      - `SIGINT` (Ctrl-C) MUST reach the foreground process group via the
        kernel's controlling-TTY plumbing — the child sees it directly
        because it owns the PTY, NOT because aish forwards it. The aish
        process MUST NOT install a `SIGINT` handler that swallows the
        signal during PTY child lifetime (the parent's default would
        kill aish itself, which is wrong).
      - `SIGWINCH` on the parent MUST trigger a `pty.InheritSize` (or
        equivalent ioctl) on the master fd so the child re-renders.
      - `SIGTSTP` (Ctrl-Z) is recorded as TODO(v0.3-jobs) — full job
        control (suspend → bg/fg) is a separate epic per GOALS §"Signal
        Handling and Job Control".
- [ ] **#56** Initial window size + ongoing resize propagation. At
      child start, the master PTY's window size MUST be initialised
      from the parent's controlling-TTY size (`pty.GetsizeFull(os.Stdin)`).
      On every `SIGWINCH` arrival, the master is re-sized to match.
      Resize is best-effort: a resize ioctl failure logs to stderr but
      does not kill the child.
- [ ] **#57** Test matrix (manual smoke + automated integration):
      - Automated: a Go test that runs `stty size` through `RunPTY`
        against a freshly allocated PTY of size `40x100`, asserts the
        child prints `40 100`. Gates `tcgetwinsz` correctness.
      - Automated: a Go test that runs `tty` through `RunPTY` and
        asserts the child reports a `/dev/ttys*` (or `/dev/pts/*` on
        Linux) device — proves the child has a controlling TTY.
      - Automated: a Go test that runs `cat` through `RunPTY`, feeds
        `hello\n`, then closes stdin, and asserts stdout receives
        `hello\r\n` (PTY line-discipline maps `\n → \r\n` on output).
      - Manual smoke (documented in PR body): `vim /tmp/x.txt`, `less
        /etc/hosts`, `top` — all render correctly, accept input, exit
        cleanly. CI cannot run these (no real controlling TTY in
        `go test`), so the automated tests cover the mechanics and the
        manual tests cover the integration.
- [ ] **Termios restoration.** When aish itself is launched under a
      real TTY (i.e. `os.Stdin` is a TTY), `RunPTY` MUST put the parent
      TTY into raw mode for the duration of the child, and MUST restore
      the original termios on child exit — including on panic and on
      child-killed-by-signal. Failure mode if skipped: aish returns to
      its REPL with the terminal in raw mode, cursor wrong, no echo,
      and the user has to type `reset` to recover. This MUST NOT
      happen.
- [ ] **No-TTY graceful degrade.** If `os.Stdin` is not a TTY (piped
      input, CI), `RunPTY` MUST still allocate a PTY for the child but
      MUST NOT attempt termios manipulation on stdin. The child still
      sees a real PTY; the parent simply does not put a non-TTY into
      raw mode.
- [ ] **Shell dispatch seam.** `shell.runExternal` decides per-pipeline
      whether to route through `exec.Run` (existing stdio path) or
      `exec.RunPTY` (new PTY path). The decision is "single command +
      parent stdin is a TTY + child binary is on the interactive list
      OR the child requested it via opt-in". For v0.2-2 the list is the
      explicit allowlist named in #57: `vim`, `vi`, `nvim`, `less`,
      `more`, `top`, `htop`, `ssh`, `az`, `nano`, `man`. Future epics
      replace the allowlist with a smarter probe (e.g.
      `isatty`-of-child detection via the cache plugin).
- [ ] **`make ci`** passes at repo root.
- [ ] **Live smoke** (REPL): `aish` → `vim /tmp/x.txt` → `:q` returns
      to the aish prompt with the cursor visible, echo on, and the
      prompt re-rendered.
- [ ] **Windows stub** compiles. On Windows `RunPTY` returns
      `errPTYUnsupported` and the dispatch seam falls back to the
      stdio path — `vim.exe` does not work yet but `aish.exe` still
      builds and runs every other command.

## Alternatives Tables

### A1 — How to allocate the PTY

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| `github.com/creack/pty` v1.1.x | Pure Go on Unix; battle-tested (used by Docker, k3s, gotty); BSD-3; one-line `pty.Start(cmd)`; window-size + InheritSize helpers built in. | Single external dep; Windows ConPTY support is incomplete in v1.1 (gated to v1.2+ or follow-up). | **Chosen** |
| In-house `syscall.ForkExec` + raw `tcsetattr` / `ioctl(TIOCPTMASTER)` | No external dep; full control over signal masks and termios. | Re-implements the hard parts (master/slave handshake, name resolution `ptsname`, owner/perm setup `grantpt`, unlock `unlockpt`) on macOS / Linux / *BSD with divergent quirks; high defect surface for low gain. | Rejected |
| No PTY — inherit aish's parent TTY directly into the child via `cmd.Stdin = os.Stdin; cmd.Stdout = os.Stdout` | Trivial — three lines. | Defeats the point: aish loses the ability to interpose (history, cache, observability) on interactive sessions; the child + aish share one TTY's process group with no boundary; and we cannot resize the child independently. Worse, aish's REPL byte-by-byte `readLine` would race the child for stdin bytes. | Rejected |

### A2 — When to allocate a PTY

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Always allocate** for every external command | Uniform behavior. | Breaks every non-interactive use of aish: pipelines (the existing stdio multi-stage `Run` is incompatible with PTY mid-stages), scripted invocations, CI runs of `aish "echo hi"`. Slower per-command (PTY allocation cost + termios round-trip). | Rejected |
| **Only when parent stdin is a TTY + command is on a curated interactive allowlist** | Matches user expectation (interactive programs get a PTY; pipelines and scripts don't). Allowlist is explicit; no probe cost on the hot path. | Allowlist is a maintenance item — new interactive binaries (`nvim`, `lazygit`, `gh`) need an entry. Mitigated by keeping the list small and visible. | **Chosen** |
| **Opt-in per command via env var** (`AISH_PTY=1 vim …`) | Most explicit; never wrong. | Wrong default — users expect `vim` to "just work" without ceremony. Fine as an escape hatch but not as the only mechanism. | Rejected as primary (kept as escape via `AISH_PTY_FORCE=1` env, low priority) |

### `exec.RunPTY` contract

```go
// RunPTY runs a single-command pipeline against the host with a real
// pseudo-terminal. The child sees a controlling TTY pointed at the
// PTY slave; the parent reads/writes the master.
//
// Constraints:
//   - len(p.Commands) MUST equal 1. Returns errPTYPipeline otherwise.
//   - stdin and stdout MUST be *os.File. The parent-side raw-mode
//     dance and SIGWINCH propagation both require a real fd. Returns
//     errPTYNeedFile if either is not an *os.File.
//
// stdin SHOULD be a TTY for the user-facing case (vim et al). If it
// isn't, RunPTY still works — the parent is simply not put into raw
// mode and there is no SIGWINCH handler. The child still gets a real
// PTY.
//
// Exit-code contract matches exec.Run: returns (exitCode, nil) for
// "child ran and exited"; (0, err) for "could not start". A child
// killed by a signal collapses to exitCode == 128+signum per POSIX.
func RunPTY(
    ctx context.Context,
    p parser.Pipeline,
    env []string,
    stdin, stdout *os.File,
    stderr io.Writer,
) (exitCode int, err error)
```

### `pty_unix.go` flow (Unix)

1. Reject `len(p.Commands) != 1` and non-`*os.File` stdin/stdout.
2. Build `osexec.CommandContext` with env.
3. `pty.Start(cmd)` — returns the master `*os.File`.
4. If stdin is a TTY (`term.IsTerminal(stdin.Fd())` — but the repo
   forbids touching `shell/internal/term`; use
   `golang.org/x/term.IsTerminal` directly inside `exec`): put it
   into raw mode and `defer` restore.
5. Set master size from stdin's size; install a `SIGWINCH` handler
   for the child's lifetime that calls `pty.InheritSize(stdin,
   master)`. Uninstall on exit (`signal.Stop` + close the channel).
6. Two goroutines: `io.Copy(master, stdin)` and `io.Copy(stdout,
   master)`. Both ride the master's lifetime — closing the master
   on child exit unblocks them.
7. `cmd.Wait()` → exit code; tear down in defer order (restore
   termios → stop SIGWINCH handler → close master).

### `pty_windows.go` stub

Returns `errPTYUnsupported`. Documented as a known limitation in
GOALS.md §"v1.0 — Windows Native". The seam exists; ConPTY plumbing
is a separate epic.

### Test strategy

- **Unit-ish integration tests** in `pty_test.go`:
  - `TestRunPTYIsATTY` — child sees `tty` resolve to a `/dev/pts/*`
    or `/dev/ttys*` device.
  - `TestRunPTYWindowSize` — child sees `stty size` return the
    correct rows/cols after we set the master to a known size.
  - `TestRunPTYStdinPassthrough` — `cat` echoes piped input back to
    stdout (via PTY line-discipline, so `\n → \r\n`).
  - `TestRunPTYRejectsPipeline` — multi-stage pipeline → `errPTYPipeline`.
  - `TestRunPTYMissingBinary` — non-existent binary surfaces a non-nil
    error (parity with `exec.Run`).
- Tests use `pty.NewPair` (or `pty.Open`) on the parent side rather
  than `os.Stdin` so they run under `go test` without a real
  controlling TTY. Skip with `t.Skip("requires PTY")` when the OS
  can't open a PTY at all (CI on a stripped image).
- Tests honor the existing `requireBinary` skip pattern from
  `exec_test.go` (we depend on real `cat`, `tty`, `stty`).
- `pty_interactive_test.go`: trivial table-driven test asserting the
  allowlist matches the names in #57 and that case-insensitive
  matching works (`Vim` and `vim` both match).

### Adversarial pass (per Code.md §11.6)

- **Termios leak on panic.** Mitigated by `defer term.Restore(fd,
  origState)` at the start of raw-mode setup, BEFORE the goroutines
  start. A panic anywhere in the body still runs the defer.
- **SIGWINCH after teardown.** Mitigated by `signal.Stop(ch)` and
  channel-close before `master.Close()`; goroutine selects on `done`.
- **Ctrl-C kills aish, not the child.** Mitigated by NOT installing
  a SIGINT handler in the parent for PTY children — the kernel
  routes SIGINT to the foreground process group via the controlling
  TTY, which is the PTY slave's group, NOT aish's group. Verified
  by inspection of `pty.Start` behavior (the child becomes the
  session leader of its own session, controlling the PTY).
- **TOCTOU on stdin TTY check.** Negligible: stdin doesn't change
  type mid-call; if it did, the worst case is we attempt raw mode
  on a non-TTY and the ioctl returns ENOTTY, which we log and
  proceed without raw mode.
- **Resize race with child startup.** The first SIGWINCH could
  arrive between `pty.Start` and `signal.Notify`. Mitigated by
  doing an explicit `pty.InheritSize` immediately after `Start`,
  THEN installing the handler — the initial size is correct even
  if no SIGWINCH ever fires.
- **Pipeline through PTY.** Out of scope: explicitly rejected with
  `errPTYPipeline`. Pipelines stay on `exec.Run`.

## Out of Scope

- Full job control (Ctrl-Z suspend, `fg`, `bg`, `jobs`) — separate
  epic per GOALS §"v0.3 — Real Shell". This plan establishes
  signal-correctness for the foreground PTY child; backgrounding is
  v0.3.
- Windows ConPTY implementation — stub returns `errPTYUnsupported`.
  Tracked in GOALS §"v1.0 — Windows Native".
- PTY pipelines (`vim file1 file2 | tee log`). The PTY model
  doesn't compose with byte-pipes the way stdio does; users who
  need this pipe to a file from inside the editor.
- Auto-detection of "is this command interactive?" — v0.2-2 ships
  an explicit allowlist; smarter probes are a separate concern.

## Dependencies

- Adds `github.com/creack/pty` to `shell/go.mod` and
  `golang.org/x/term` (already an indirect dep via other modules;
  promote to direct). Both are BSD-3 / BSD-style and Apache-2 (term).
- No changes to `libs/proto`, `plugins/**`, or `shell/internal/term/`
  (TL_UI's domain — parent-side TTY input, distinct from this
  child-side work).

## Backward Compatibility

- `exec.Run` signature is unchanged; existing callers in `shell.go`
  for non-interactive pipelines continue to work bit-for-bit.
- `shell.runExternal` only diverts to `exec.RunPTY` when the
  conditions hold (single command + interactive name + parent TTY).
  Every test in `exec_test.go` and `shell_test.go` keeps its
  current behavior.
- No changes to user-visible flags, env vars, or config files.
