## Acceptance Criteria

- [ ] **#30** Structured event schema (JSON, append-only) covering `id, ts,
      kind, command, cwd, exit_code, duration_ms, affected[]` where each
      `affected` is `{path, op, snapshot_dir|nil}`. Schema is in code (Go
      struct + JSON tags) and is the source of truth for both row encoding
      and the on-disk JSONL mirror.
- [ ] **#31** Event log at `~/.aish/history.db` (SQLite, `journal_mode=WAL`,
      `busy_timeout=5000`). Two tables: `events` (one row per shell
      command, JSON payload) and `snapshots` (one row per snapshotted file,
      pointing at the on-disk copy). Append-only by convention (no UPDATE
      paths in the writer except `events.exit_code` post-execution).
- [ ] **#32** Pre-execution detector inspects `parser.Pipeline` and flags
      any command name in a curated destructive list (`rm`, `rmdir`,
      `unlink`, `truncate`, `shred`, `mv` *(only when target exists)*, `dd
      of=`). False positives are acceptable (extra snapshot work); false
      negatives MUST be impossible for the curated list.
- [ ] **#33** Pre-execution snapshot: for each path in
      `expandTargets(pipeline)`, copy the bytes verbatim to
      `~/.aish/snapshots/<UTC-ISO-8601>-<sha1(path)>/<basename>` BEFORE
      the command runs. Directories are walked. Each file gets its own
      snapshot row. The original mtime is preserved on the snapshot.
- [ ] **#34** Snapshot size limit `~/.aish/config.toml` key
      `history.snapshot_max_bytes` (default `100MB`). Files larger than
      the limit are recorded in the event log with `op=skipped,
      reason=oversize` and NOT copied. The user sees a stderr notice.
- [ ] **#35** `aish undo` restores from the most-recent event whose
      `kind=snapshot` and `exit_code != null`. Restores every snapshotted
      path; verifies hashes after copy-back; refuses if a target path
      currently has different bytes (conflict resolution is out of scope —
      surface the conflict, do nothing).
- [ ] **#36** `aish restore <path>` restores the most-recent snapshot of
      `<path>` regardless of which event it came from. Useful when an
      earlier-than-last destructive op is the one the user wants reverted.
- [ ] **#37** Pre-snapshot path filter honors a built-in
      `.gitignore`-style pattern list (defaults: `.git/`, `node_modules/`,
      `vendor/`, `dist/`, `build/`, `target/`, `__pycache__/`, `*.tmp`,
      `*.log`, `.cache/`). Skipped paths get `op=skipped, reason=ignored`
      in the event log. The filter applies on the *write* side — at
      snapshot time, not at restore time — so an `aish restore` of an
      ignored path returns "no snapshot recorded."
- [ ] **`make ci` passes** at repo root.
- [ ] **Live smoke** (`HOME=$(mktemp -d) ./shell/dist/aish` →
      `echo test > /tmp/x; rm /tmp/x; undo; cat /tmp/x` prints `test`).

## Interceptor seam (designed for TL2's PostCommand observer)

```go
// shell/internal/shell/interceptor.go (new file, TINY)
type Interceptor interface {
    Before(pipeline *parser.Pipeline, line string) error
    After(pipeline *parser.Pipeline, line string, exitCode int, dur time.Duration)
}
```

History registers as one Interceptor. Telemetry (TL2's domain) will register
as a separate Interceptor on the same slice. The Shell holds
`interceptors []Interceptor` and iterates twice — once before exec, once
after. TL2's PR adds an entry; TL2 does NOT touch the seam itself.

A `Before` error MUST NOT abort the command (per acceptance: "false
positives are fine"); it is logged to stderr and execution continues. The
discipline is "snapshot is best-effort, command is mandatory."

## Alternatives Table — Snapshot storage

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **A. Raw file copies under `~/.aish/snapshots/<UTC>-<sha1(path)>/<basename>`** | Simple. Human-inspectable. Restore is `cp -p`. No bespoke on-disk format. Cross-platform trivially. | No deduplication — a 100MB file deleted twice costs 200MB. | **CHOSEN** |
| **B. Content-addressed store (`~/.aish/cas/<sha256-prefix>/<sha256>`) with snapshot rows pointing at the digest** | Automatic dedup; same bytes only stored once. Aligns with "civilization-grade" structural discipline. | Significantly more complex: garbage collection on undo invalidation, reference counting, atomic write into CAS. Out of scope for v0.1 "viral demo." | Rejected for v0.1; revisit in v0.3-4 ("History Engine Maturity") when modification snapshots land. |
| **C. SQLite BLOB column** | Single-file store. Atomic with the event row. | Heavy 100MB writes inside SQLite are slow and bloat the DB; restore requires streaming out of SQLite which is awkward for large files. | Rejected. |

Choice: **A** for v0.1. CAS becomes the natural migration when v0.3-4
brings modification snapshots and the dedup pressure actually materializes.

## Alternatives Table — gitignore matcher

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **A. Vendor `github.com/sabhiram/go-gitignore`** | Battle-tested. Handles full gitignore semantics (negation, anchored patterns, `**` globs). | One more dep; BSD-3 (license OK). | Considered. |
| **B. Roll minimal matcher (literal dir-prefix + `*.ext` suffix)** | Zero new deps. Covers the documented v0.1 defaults exactly. | Will NOT grow to cover full gitignore semantics — limits future extensibility. | **CHOSEN for v0.1** — the default pattern list is small, the matcher is ~60 LOC, future expansion is a single-package replacement. |

Choice: **B** for v0.1. The matcher is internal to `history/` and can be
swapped to `sabhiram/go-gitignore` in a single-file PR when the patterns
grow beyond the v0.1 defaults.

## Test Strategy

- **Unit per file**, table-driven where helpful.
- **`detect_test.go`** — pipeline fixtures across `rm`, `rmdir`, `mv`, `dd
  of=`, harmless commands (`ls`, `cat`, `grep`); verify
  `IsDestructive` boolean and `TargetPaths` slice.
- **`ignore_test.go`** — match `node_modules/foo.js`, `.git/HEAD`,

  `os.Remove(original)` → `Restore()` → `bytes.Equal`. Oversize file path:
  set limit to 100B, snapshot 200B file, assert `op=skipped`.
- **`store_test.go`** — `Append` then `LatestRestorable` returns the
  newest `exit_code!=null` event; `SnapshotsForPath("/tmp/x")` returns
  most-recent snapshot for that path.
- **`interceptor_test.go`** — full Before/After cycle with an in-memory
  store + tempdir snapshot root.
- **`builtin_undo_test.go`** — set up an event with a snapshot in
  tempdir, delete the live file, run the built-in, assert restored bytes.
- **`shell_acceptance_test.go`** — extend with one e2e: REPL
  `echo X > /tmp/aish-T/x; rm /tmp/aish-T/x; undo; cat /tmp/aish-T/x`

## Risks

| Risk | Mitigation |
|---|---|
| Path-resolution mistakes (relative vs absolute) leave snapshot pointing at the wrong file | `TargetPaths` canonicalizes via `filepath.Abs` against the shell cwd before snapshotting. |
| Race between detector parse and child exec — file gone between snapshot and `rm` | Snapshot is a content-copy; the `rm` deleting a file already snapshotted is the *intended* sequence. Re-snapshot on every Before; no caching. |
| SQLite WAL contention with cache.db | Separate file (`history.db`), separate connection. WAL is per-file. |
| Snapshot dir balloons | 100MB per-file default + ignore filter keep growth bounded for v0.1. Pruning policy is deferred (v0.3-4). |
| Undo on directory delete vs file delete | v0.1 scope = files. Directory restores walk the directory's snapshot subtree; if a single dir-snapshot row exists per child file, restore reconstructs the tree by re-creating intermediate dirs. |
