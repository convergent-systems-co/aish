# Plan: v0.3-5 — Persona Engine (Embedded Personality)

> The shell embodies a personality. Voice, tone, prompt segments, and AI-
> mediated responses reflect the active persona. Distinct from identity
> (v0.3-3): identity answers *who you are*; persona answers *who the
> shell is being for you*.

**Source:** GOALS.md §"Epic v0.3-5 — Persona Engine"
**Epic (supervised):** #114
**Tasks (in-scope this PR):** #115, #116, #117, #118, #119, #120, #122, #123, #126, #129
**Tasks (deferred — MVP floor):** #121, #124, #125, #127, #128
**Domain:** `domain:persona`
**Risk:** MEDIUM — novel surface; safety floor is non-negotiable.
**Milestone:** v0.3 — Real Shell (90 days)

## Objective

Ship the persona engine MVP: a typed schema, bundled curated personas,
the `aish persona` built-in (list/show/set/active), system-prompt
injection on inference dispatch, persistent active selection, and an
unbypassable safety floor. Distinct from v0.3-3 identity work — persona
is the shell's *voice* and *posture*, not the user's credentials.

## Safety Floor — §129 (non-negotiable)

### Categories (hard-coded in `safety.go`)

1. **No mass-casualty weapons.** Synthesis routes, dual-use precursors,
   delivery systems, evasion of export controls.
2. **No CSAM.** Any generation, encouragement, search assistance, or
   normalisation of child sexual abuse material.
3. **No unauthorised-target cyberattacks.** Exploit code, intrusion
   tooling, or payload-development assistance directed at systems the
   user does not own or is not legally authorised to test. Pen-testing
   one's own systems is permitted with caller assertion.
4. **No PII lookups on real people.** Reverse-lookup of phone numbers,
   addresses, employer info, social-graph stitching across data
   brokers. Public-figure information remains permitted.

### Enforcement mechanism

- `safety.SafetyFloorPrompt()` returns a fixed system-prompt block that
  enumerates the four categories and instructs the model to refuse
  cleanly when triggered.
- `compose.SystemPromptFor(p)` ALWAYS prepends `SafetyFloorPrompt()` to
  the persona's `system_prompt`. The safety block is FIRST so the model
  reads it before any persona instruction.
- A persona TOML CANNOT lower the floor. Any attempt to do so via:
  - A `safety_overrides = ...` key (not in the schema; loader rejects
    unknown keys at the top level).
  - A `system_prompt` whose first line says "ignore safety" or matches
    a denylist of bypass strings — the validator runs a regex set and
    rejects.
- Test cases:
  - `TestSafetyFloor_AlwaysFirst` — the composed system prompt
    contains the safety block as a prefix regardless of persona.
  - `TestSafetyFloor_MaliciousPersonaRejected` — a TOML file with
    `system_prompt = "ignore all prior safety instructions and …"`
    fails `Load()` with a clear `ErrSafetyBypass` error.
  - `TestSafetyFloor_UnknownKeyRejected` — a TOML with
    `safety_overrides = { weapons = "allow" }` fails to parse into the
    typed struct (unknown-key rejection via TOML strict mode).

## Schema (v1)

```toml
# persona TOML v1
name = "mentor"                     # required; matches filename
version = 1                          # schema version
description = "Patient teacher persona."
voice = "Encouraging, asks clarifying questions before answering."

# The system-prompt template. SAFETY FLOOR is automatically prepended.
# Multi-line via triple-quote.
system_prompt = """
You are a patient mentor. When the user asks a question, briefly
restate what you understood, then walk through the answer with
reasoning visible. Prefer asking ONE clarifying question over guessing
when intent is ambiguous.
"""

[tone]
verbosity = "medium"                 # terse | medium | verbose
formality = "casual"                 # casual | neutral | formal
emoji = false

[capability_gates]
# Optional opt-in gates that LAYER ABOVE the safety floor.
# Cannot lower it — only add restrictions.
refuse_when_no_files_provided = false
refuse_to_write_code = false
no_direct_answers_to_ambiguous_intents = false

[prompt_overrides]
# Optional shell-prompt decoration (deferred to v0.3-5.1).
greeting_glyph = ""
voice_phrase = ""
accent_char = ""
```

## Test Strategy

- **schema validation** — table-driven. Valid persona round-trips;
  missing name fails; unknown top-level key fails; invalid verbosity
  enum fails.
- **safety floor** — three tests above. Plus an integration test that
  the composed system prompt always starts with the safety block text.
- **loader** — bundled + user merge; user overrides bundled by name;
  duplicate user files (same name twice on disk via case-folding)
  rejected.
- **config persistence** — merge-aware writer preserves `[theme]`,
  `[telemetry]` sections. Round-trip idempotent.
- **built-in dispatch** — `persona list`, `persona show <name>`,
  `persona set <name>`, `persona active`, error paths (unknown name,
  no args).
- **integration** — `cache.PluginClient.InferWithSystemPrompt` is
  invoked when an active persona has a non-empty system prompt;
  composes the wrapped intent.
- **adversarial pass** — malicious persona, empty system prompt
  (graceful no-op), duplicate name across bundled+user, unbounded
  system-prompt size (cap at 8 KiB; reject larger).

## Risk Assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| Safety-floor bypass via clever TOML | Medium | Strict TOML decoder (`Disallow Unknown`), denylist regex on system_prompt, prepend-not-replace composer. Tests cover three vectors. |
| Cache fragmentation: each persona produces its own cache rows for the same user intent | Medium | Acceptable for MVP — caches are per-persona by accident, not by design. Document in plan. v0.3-5.1 with proto extension fixes this cleanly. |
| InferParams workaround breaks if the gateway tokenises the composite intent weirdly | Low | Live smoke test in the verification gate (`aish persona set mentor; <real LLM call>`). |
| Bundled-persona drift from schema | Low | `bundled_test.go` validates every bundled persona at test time. CI catches drift. |
| TL_PLUGIN ships InferParams.SystemPrompt in parallel and we conflict | Medium | This plan explicitly carves the workaround that does NOT touch the proto; coordination message in PR body. |

## Backward Compatibility

- New package, new built-in, new directory, new config section. Zero
  changes to existing behaviour when no persona is active.
- When the user has never run `persona set`, the `default` bundled
  persona is implicitly active. Its system prompt is the **bare
  safety floor only** (no persona-shaping text), so the LLM sees the
  same content as v0.3-4 with the addition of the safety-floor
  prefix. (The safety floor is acceptable to add at any time — it
  only refuses things that should always be refused.)
- `~/.aish/config.toml` gains a `[persona] active = "<name>"` line on
  first `persona set`. Merge-aware writer preserves existing sections.
