# Plan: v1.0-1 — Windows Build Pipeline

> Land the build-pipeline scaffold for Windows: cross-compiled Win amd64 /
> arm64 binaries from CI, a signed-installer manifest (winget the primary
> channel, MSI/WiX a follow-up), Windows-Terminal compatibility checks
> baked into the dev profile, and PowerShell-interop documentation +
> profile snippets for hybrid users. Pure-Go promise of the project means
> no CGO for the shell — `GOOS=windows go build` from macOS/Linux works
> today, so the work is mostly Makefile targets, a GitHub Actions workflow,
> and packaging recipes.

**Source:** GOALS.md §"Epic v1.0-1 — Windows Build Pipeline" (lines 1005-1010)
**Epic (supervised):** #130
**Tasks (in scope this PR):** #131 (cross-compile from CI),
#132 (signed installer — winget recipe + MSI/WiX manifest, signing
deferred per §3 alt-table), #133 (Windows Terminal compat),
#134 (PowerShell compat for hybrid users).
**Domain:** `domain:build`
**Risk:** Medium per GOALS.md. The build-pipeline surface is mechanical
(Makefile, YAML, JSON, XML); the real Windows-runtime risk lives in
v1.0-2 / v1.0-4 / v1.0-5 (sibling TLs).
**Milestone:** v1.0 — Windows Native (120 days)

## Acceptance Criteria

### MVP-floor (shipping this PR)

- [ ] **Windows cross-compile from `make build-windows` (#131).** Top-level
      `make build-windows` produces `dist/aish-windows-amd64.exe`,
      `dist/aish-windows-arm64.exe`, `dist/aish-inference-cloud-windows-*.exe`,
      and `dist/aish-plugin-windows-*.exe` — i.e. all four release binaries
      across both Windows architectures. Implementation delegates to the
      existing `build-all` plumbing in `shell/Makefile` and
      `plugins/cloud/Makefile` via two new per-module targets:
      `build-windows` (both arches) and a generic `build-windows-%`
      pattern rule for one-arch builds (`build-windows-amd64`,
      `build-windows-arm64`). Top-level `make build-windows` aggregates
      the per-module outputs into `dist/` at the repo root for downstream
      packaging.
- [ ] **`make release-windows` produces SHA256-sealed artifacts.**
      `make release-windows` runs `build-windows` then writes
      `dist/aish-windows-amd64.exe.sha256` etc. plus a top-level
      `dist/aish-windows-MANIFEST.txt` listing every file with size +
      sha256. This is the artifact set the installer recipe consumes.
- [ ] **GitHub Actions workflow `.github/workflows/release-windows.yml`
      (#131).** Triggers on `push: tags: ['v*']` and
      `workflow_dispatch`. Runs on `ubuntu-latest` (cross-compile,
      keeps the per-minute cost low) with `actions/setup-go@v5` pinned
      to the repo's Go version. Steps: checkout, setup-go,
      `make release-windows`, upload `dist/aish-windows-*` as a
      workflow artifact, and (on tag pushes only) attach the artifacts
      to a GitHub Release via `softprops/action-gh-release@v2`. Signing
      is **disabled by default** (gated behind `secrets.WINDOWS_CERT_PFX`
      — workflow no-ops the signing step when the secret is absent
      so forks and feature branches don't break).
- [ ] **Workflow lints cleanly.** `actionlint` exits 0 against the new
      YAML (verified locally; CI step also runs `actionlint` via
      `rhysd/actionlint@v1` so PRs catch regressions). When `actionlint`
      is unavailable locally, `yamllint` fallback is documented in the
      scripts dir.
- [ ] **Stub PowerShell signing script `scripts/win/sign.ps1` (#132).**
      Signs an `.exe` or `.msi` using `signtool.exe` with the cert
      identified by `$env:WINDOWS_CERT_THUMBPRINT` (the production
      path) OR `$env:WINDOWS_CERT_PFX` + `$env:WINDOWS_CERT_PASSWORD`
      (the PFX path used by CI). The script is **dormant** in this PR:
      no cert is checked in, no secret is consumed; `make ci` does not
      invoke it. The workflow calls it conditionally (skip when the
      secret is empty). This preserves `Common.md P4 — No Secrets In
      Artifacts` while landing the structure the prod cert will later
      slot into.
- [ ] **winget manifest stub `data/install/windows/winget/aish.yaml`
      (#132).** Skeleton winget manifest (`PackageIdentifier:
      ConvergentSystems.aish`, `Publisher`, `License`, `InstallerType:
      portable`, per-arch `InstallerSha256` placeholders, `Tags`,
      `ShortDescription`). The release workflow does NOT publish to
      winget-pkgs in this PR — publishing requires a per-release PR to
      microsoft/winget-pkgs, which is a deliberate human step.
- [ ] **WiX MSI manifest stub `data/install/windows/wix/aish.wxs`
      (#132).** Minimal WiX 4 source: product GUID,
      per-arch component group, `aish.exe` installed to
      `%LocalAppData%\Programs\aish\`, PATH augmentation via
      `Environment` element, uninstall entry. MSI build is **not** wired
      into `make build-windows` (WiX requires Windows + .NET tooling
      not present on the cross-compile host); the script
      `scripts/win/build-msi.ps1` produces it on a Windows runner only.
      The workflow's MSI job is `if: false` for now (lights-off) with a
      `TODO(v1.0-1b)` note pointing to the follow-up task.
- [ ] **Windows Terminal profile fragment `data/install/windows/wt/profile.json`
      (#133).** A JSON fragment users can paste into
      `%LocalAppData%\Packages\Microsoft.WindowsTerminal_*\LocalState\
      settings.json` under `"profiles" -> "list"`. Includes:
      `commandline = "aish.exe"`, `name = "aish"`, `icon` reference
      (relative path commented), `startingDirectory =
      "%USERPROFILE%"`, `colorScheme = "Campbell"` (default WT scheme;
      no aish-specific scheme this PR — defer to v1.0-3 with the
      script-translation epic).
- [ ] **Windows Terminal compat smoke `scripts/win/wt-compat-check.ps1`
      (#133).** Runs from a PowerShell session inside Windows Terminal:
      detects `$env:WT_SESSION` (Windows Terminal sets this), verifies
      `$env:TERM_PROGRAM` and `$Host.UI.RawUI.WindowSize` resolve
      sensibly, runs `aish.exe --version` and captures the output, and
      writes a TAP-style report. The script is **documentary**; it does
      not run in CI (no Windows runner this PR), but the workflow's
      `windows-smoke` job stub (`if: false`) holds the call so wiring
      a Windows runner later is one-line.
- [ ] **PowerShell profile snippet `data/install/windows/powershell/Profile.ps1`
      (#134).** Drop-in users append to their `$PROFILE`. Defines:
      - `aish` PowerShell function that pipes-through stdin/stdout to
        `aish.exe` (so PowerShell pipelines `Get-ChildItem | aish ...`
        feed bytes, not objects).
      - `Invoke-Aish` advanced function with proper
        `[CmdletBinding()]` and `param([Parameter(ValueFromPipeline)]$Input)`
        so object pipelines convert via `Out-String` before reaching
        aish. Hybrid users keep object semantics on the PowerShell
        side and stream semantics on the aish side.
      - `Set-Alias ai aish` for ergonomics.
      - `# aish-managed: keep this block intact for upgrades` marker
        comment for future install-time edits.
- [ ] **PowerShell snippet has a fixture test.** `scripts/win/test/
      Profile-Test.ps1` (Pester-flavored; runs under PowerShell 7+ on
      any platform) asserts: file parses (`Invoke-Expression (Get-Content
      Profile.ps1 -Raw)` doesn't throw), the four symbols (`aish`,
      `Invoke-Aish`, `ai`, marker comment) exist after dot-sourcing.
      Pester is the de-facto PowerShell test framework and ships on
      every modern PowerShell — the runbook documents the one-line
      install (`Install-Module Pester -Force`) for users who lack it.
- [ ] **README addendum `data/install/windows/README.md`.** One page,
      audience = "Windows user installing aish from a v1.0+ release."
      Covers: download → verify SHA256 → unzip → PATH addition (one
      PowerShell line) → Windows Terminal profile paste → PowerShell
      profile snippet append → first-run. ≤ 200 lines.
- [ ] **`make ci` passes** from the worktree. (Pre-existing modules
      unchanged; only Makefile additions land in this PR.)
- [ ] **`make build-windows` exit 0 on macOS/Linux**, produces the
      eight expected files (`dist/aish-windows-{amd64,arm64}.exe`,
      `dist/aish-inference-cloud-windows-{amd64,arm64}.exe`,
      `dist/aish-plugin-windows-{amd64,arm64}.exe`, two sha256s).
- [ ] **`actionlint .github/workflows/release-windows.yml` exit 0**
      (verified via `make lint-workflows` — a new top-level target that
      runs `actionlint` if installed and prints a skip-message if not).

### Deferred (acknowledged stubs / follow-up issues)

- [ ] **Production code signing.** No EV / OV certificate exists yet.
      The PR lands the signing-script + workflow plumbing such that
      adding `secrets.WINDOWS_CERT_PFX` enables it without a code
      change. **Deferred to v1.0-1b** (issue to be filed at PR merge:
      "Procure Windows code-signing certificate and enable signtool
      in release-windows.yml"). The dormant path is documented in
      `data/install/windows/README.md` under "Verifying authenticity"
      — users get sha256 verification today, signature verification
      once the cert arrives.
- [ ] **winget-pkgs PR automation.** Publishing to microsoft/winget-pkgs
      is a per-release `wingetcreate` invocation that opens an
      upstream PR. **Deferred to v1.0-1c** — the manifest format is
      committed today; the bot wiring is a follow-up so the first PR
      to winget-pkgs can be reviewed by a human.
- [ ] **MSI build job in CI.** Requires a `windows-latest` runner with
      WiX 4 SDK installed; the job lights up when (a) the cert lands
      and (b) we accept the per-minute Windows-runner cost. **Deferred
      to v1.0-1b** alongside signing.
- [ ] **Chocolatey package.** Considered in §3 alt-table and rejected
      for v1.0. The winget channel is the modern Windows package
      manager (shipped in Windows 11 by default); chocolatey is
      addable later if community demand surfaces. **Not deferred —
      explicitly out of scope.**
- [ ] **Windows runner CI smoke job.** `release-windows.yml` carries
      a `windows-smoke` job stub (`if: false`) that the v1.0-1b PR
      flips on. The stub keeps the wiring honest so the follow-up
      doesn't reinvent it.

### Out of scope (deliberate exclusion, not deferral)

- Touching `shell/internal/**`, `plugins/cloud/internal/**`, or
  `libs/proto/**`. Those packages belong to TL_WIN_OS / TL_WIN_SECRETS
  / TL_WIN_LOGIN per the partition. The build pipeline consumes the
  binaries they produce; it does not modify them.
- ConPTY integration, Windows Credential Manager bindings, Win32 CGO,
  PowerShell script reader. All v1.0-2 / v1.0-3 / v1.0-4 / v1.0-5
  surface — owned by sibling TLs in parallel worktrees.
- Removing the existing `windows/amd64` and `windows/arm64` entries
  from each module's `PLATFORMS` list. The new top-level targets are
  **additive**; the existing `build-all` continues to produce Windows
  binaries today, just without the dedicated `build-windows` /
  `release-windows` ergonomics.

## Rationale + Alternatives Tables

### Alternative: installer channel

| Alternative                                     | Pros                                                                                                                | Cons                                                                                                                                                  | Verdict |
|---|---|---|---|
| **winget (primary) + WiX/MSI manifest stub for future** | Winget is shipped in Windows 11 by default; the most native modern channel. WiX MSI gives an offline-installable artifact for enterprise/airgapped. | Two manifests to keep aligned. MSI build needs a Windows runner.                                                                                  | **Chosen** |
| MSI only (WiX)                                  | One installer format; works on every Windows from 7 forward.                                                         | No discoverability — users have to find the download. Doesn't match where Windows is heading (winget).                                                | Rejected |
| winget only                                     | One channel; simplest manifest.                                                                                       | No artifact for offline / enterprise / pre-Win11 installs. winget itself is a thin wrapper over the upstream binary, so we still need a portable zip. | Rejected |
| Chocolatey                                      | Established community; large existing user base.                                                                      | Third-party tool; users have to install choco first. Microsoft is moving the platform toward winget.                                                  | Rejected |
| All four (winget + MSI + choco + scoop)         | Maximum reach.                                                                                                        | Four manifests, four signing flows, four release paths to test. Premature optimization for a project pre-v1.0-launch.                                  | Rejected |

Resolution: **winget as the published channel; WiX MSI manifest committed but build-job dormant** until v1.0-1b lights up the Windows runner. Chocolatey explicitly rejected; scoop unconsidered (low volume).

### Alternative: code-signing certificate source

| Alternative                                     | Pros                                                                                            | Cons                                                                                                                                      | Verdict |
|---|---|---|---|
| **Deferred (no cert; sha256 manifest only)**    | No procurement blocker; ships v1.0-1 today. SHA256 manifest gives integrity verification.        | Windows SmartScreen warns on unsigned `.exe`. Users see a "Windows protected your PC" dialog the first time.                              | **Chosen** |
| Self-signed dev certificate                     | Free, immediate.                                                                                 | Still triggers SmartScreen (no chain to trusted root). Worse: looks like a real signature to inexperienced users — false reassurance.     | Rejected |
| OV (Organization Validation) cert from CA       | $200-400/year; eliminates SmartScreen after reputation builds.                                   | Reputation takes ~thousands of downloads. Procurement cycle. Not ready in 30 days.                                                        | Rejected (now) |
| EV (Extended Validation) cert from CA           | Immediate SmartScreen bypass on day one; hardware token requirement (HSM).                       | $400-700/year; HSM key ceremony; CI signing becomes a remote-signing flow not a `signtool /f cert.pfx` flow.                              | Rejected (now) |

Resolution: **Defer signing.** Wire the signing path so adding a cert is a secrets-only change. File the procurement task as v1.0-1b. Document SHA256 verification as the day-one trust model in the Windows README. (Rationale: signing without a real cert is worse than not signing — see false-reassurance row.)

### Alternative: CI runner choice

| Alternative                                     | Pros                                                                                            | Cons                                                                                          | Verdict |
|---|---|---|---|
| **`ubuntu-latest` for cross-compile; Windows runner deferred to v1.0-1b** | Pure-Go promise means Windows binaries cross-compile cleanly. ~1/8 the per-minute cost of windows-latest. | Can't run Windows smoke tests in CI. Can't build the MSI in CI.                                | **Chosen** |
| `windows-latest` for everything                 | Native build environment. WiX MSI + signing work in-job. Smoke tests can run.                    | ~$0.016/min vs $0.008/min for ubuntu; 8× cost for the build step. WiX-on-windows-runner setup is non-trivial. | Rejected (now) |
| Self-hosted Windows runner (scaleway)           | Cost control at scale; persistent tooling.                                                       | Maintenance burden; security exposure (self-hosted runners on public repos = supply-chain risk). | Rejected |
| Matrix: ubuntu for build, windows for smoke     | Best of both.                                                                                    | Two jobs, two cost lines. Worth it once we have something to smoke-test (post-v1.0-2).       | Deferred to v1.0-1b |

Resolution: **`ubuntu-latest` cross-compile this PR**. The `windows-smoke` job is committed as a `if: false` stub. v1.0-1b flips the flag.

### Alternative: PowerShell interop surface

| Alternative                                     | Pros                                                                                            | Cons                                                                                                                                                  | Verdict |
|---|---|---|---|
| **Profile snippet (`aish` function + `Invoke-Aish` advanced function)** | Drop-in for users; no new module to install. Handles both byte-stream and object-pipeline cases. | Snippet drift over time — users have to re-paste on upgrade. We mark the block with `# aish-managed:` so a future installer can update it in place.    | **Chosen** |
| Ship a PowerShell module (`Aish.psd1` + `Aish.psm1`) | Versioned, installable via `Install-Module`, discoverable in PSGallery.                          | Module publication requires PSGallery account + signing. Module imports add startup cost to every PowerShell session for users who don't want aish loaded by default. | Rejected (now) |
| No PowerShell integration (users invoke `aish.exe` directly) | Simplest.                                                                                       | Object pipelines feed binary garbage into aish's stdin. Misses the hybrid-user value prop of #134.                                                    | Rejected |

Resolution: **Profile snippet today; PSGallery module deferred** to whenever a hybrid-user community surfaces.

## Approach (numbered steps)

1. Add `build-windows` / `build-windows-amd64` / `build-windows-arm64`
   to `shell/Makefile` as thin wrappers over the existing `build-all`
   per-arch logic. Same in `plugins/cloud/Makefile`.
2. Add `build-windows` and `release-windows` to the top-level
   `Makefile` — both delegate to the per-module targets and aggregate
   outputs into `dist/` at the repo root. `release-windows` additionally
   writes a `dist/aish-windows-MANIFEST.txt` and per-file sha256s.
3. Add `lint-workflows` to the top-level Makefile — runs `actionlint`
   if present, prints a skip-message otherwise.
4. Run `make build-windows` locally on macOS — verify all eight files
   land in `dist/`. (Verification gate #2.)
5. Write `.github/workflows/release-windows.yml` — checkout, setup-go,
   `make release-windows`, upload artifacts, conditional release on
   tag. Add the `windows-smoke` and `windows-msi` jobs as `if: false`
   stubs.
6. Run `actionlint .github/workflows/release-windows.yml` — fix
   anything it flags. (Verification gate #3.)
7. Write the four PowerShell scripts under `scripts/win/` — sign,
   build-msi, wt-compat-check, test/Profile-Test. Each has a header
   comment explaining its dormancy / live status.
8. Write the four `data/install/windows/` recipe files — winget yaml,
   WiX wxs, wt profile json, PowerShell Profile.ps1.
9. Write the `data/install/windows/README.md` user-install doc and
   `scripts/win/README.md` operator doc.
10. Run `make ci` — confirm the existing modules' CI is unaffected.
    (Verification gate #1.)
11. Commit per `Code.md §11.2` — one logical change per commit. Plan
    → Makefile targets → workflow YAML → packaging recipes → docs.
12. Open PR with `Closes #130 #131 #132 #133 #134`.

## Testing strategy

**Build-system tests** (shell-level, run from the worktree):

- `make build-windows` on macOS produces all eight expected files.
- `make release-windows` produces the eight files **plus** eight
  `.sha256` files **plus** `dist/aish-windows-MANIFEST.txt`.
- `file dist/aish-windows-amd64.exe` reports `PE32+ executable
  (console) x86-64, for MS Windows`.
- `file dist/aish-windows-arm64.exe` reports `PE32+ executable
  (console) Aarch64, for MS Windows`.

**Workflow lint** (CI + local):

- `actionlint .github/workflows/release-windows.yml` exit 0.
- `actionlint .github/workflows/*.yml` exit 0 (no regression on the
  existing two workflows).

**PowerShell snippet fixture** (Pester, run under `pwsh` on any host):

- `pwsh scripts/win/test/Profile-Test.ps1` — asserts the profile
  parses, defines `aish`, `Invoke-Aish`, alias `ai`, and contains the
  managed-block marker. The script exits 0 on success, ≥1 on failure,
  so it can wire into CI once a `pwsh` step is added (deferred to
  v1.0-1b along with the Windows runner).

**Manifest sanity** (manual + readable in PR):

- winget manifest YAML parses (`pip install winget-create` and run
  `wingetcreate validate` locally — documented in
  `scripts/win/README.md`, not gated in CI).
- WiX `aish.wxs` is well-formed XML (`xmllint --noout aish.wxs` — also
  documented).

**No Go tests change in this PR.** The build pipeline doesn't touch
Go source; the existing `make test` continues to cover the
shell/plugins behavior.

## Risk assessment

| Risk                                                                        | Mitigation                                                                                                                                                          |
|---|---|
| Cross-compiled `aish-windows-*.exe` triggers Windows SmartScreen            | Documented in `data/install/windows/README.md`. Defer to v1.0-1b for a real cert. SHA256 manifest is the day-one trust signal.                                       |
| Workflow YAML drifts vs. `actionlint` updates                                | `make lint-workflows` runs locally; the workflow also runs `actionlint` as a job step so PRs catch regressions before merge.                                        |
| WiX manifest stub diverges from `aish.exe` reality (wrong product GUID, etc.) | The WxS file carries a `<!-- TODO(v1.0-1b): regenerate GUID and bind to real exe -->` comment; the build-msi.ps1 script is `if: false`-gated so a stale manifest never reaches users. |
| PowerShell profile snippet collides with user's existing `$PROFILE`         | `# aish-managed:` marker comment delineates the block; the README instructs users to append, not overwrite. Future installer can in-place-edit by anchoring on the marker. |
| Winget manifest's sha256 placeholders are wrong at release time             | Release workflow's tag-build step generates the real sha256, writes them into `data/install/windows/winget/aish.yaml` (in-memory; not committed back), and uploads the resolved manifest as a release artifact. The committed file is the template; the released file is hydrated. (Implementation lives in a small Make recipe — committed as `scripts/win/hydrate-winget.sh`.) |
| `windows-smoke` `if: false` stub rots                                       | The stub holds a `TODO(v1.0-1b)` comment and a one-line job body. Re-enabling is `s/if: false/if: ${{ matrix.os == 'windows-latest' }}/`.                            |
| Signing script's secret handling leaks the cert path or password            | `scripts/win/sign.ps1` uses `$env:WINDOWS_CERT_PFX` and `$env:WINDOWS_CERT_PASSWORD` only — never echoed, never written to logs. Per `Common.md §4.1`.               |

## Dependencies

- **Go toolchain** — already used by the existing `build-all` path.
  No new version requirement (cross-compile to Windows has been
  supported since Go 1.0).
- **`actionlint`** — optional, recommended. `brew install actionlint`
  on the dev host; `rhysd/actionlint@v1` in CI.
- **`signtool.exe`** — Windows-only, used by `sign.ps1`. Dormant this
  PR; lights up when the cert lands.
- **WiX 4 SDK** — Windows-only, used by `build-msi.ps1`. Dormant this
  PR; lights up with the MSI job.
- **Pester** — PowerShell test framework. Documented as a one-line
  install in `scripts/win/README.md`; not gated in CI.
- **`wingetcreate`** — Windows-only, used for the winget-pkgs PR
  flow. Out of scope this PR (manual at first release per §3
  alt-table).

**No upstream blockers.** All four sibling TLs (TL_WIN_OS,
TL_WIN_SECRETS, TL_WIN_LOGIN, and TL_WIN_BUILD) produce binaries that
already cross-compile via the existing `build-all` rule. This PR
adds ergonomics + packaging — it does not gate on their runtime work.
