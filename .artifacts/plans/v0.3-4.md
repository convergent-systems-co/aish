# Plan: v0.3-4 — History Engine Maturity

> Wave 4 of v0.3. v0.1-4 shipped event-log + snapshots + undo/restore.
> v0.3-4 matures the engine: signing for tamper evidence, a queryable
> `history` built-in with FTS5 search, modification snapshots, move /
> rename tracking, and checkpoint / rollback for named recovery points.

**Source:** GOALS.md §"Epic v0.3-4 — History Engine Maturity"
**Epic:** #107
**Tasks:** #108, #109, #110, #111, #112, #113
**Domain:** `domain:history`
**Risk:** MEDIUM
**Milestone:** v0.3 — Real Shell (90 days)

## Objective

Take the v0.1-4 history engine from "audit log + undo last destructive
op" to a real reversibility surface: every event is signed, the history
is queryable, modifications are snapshottable (not just deletes), moves
preserve provenance, and the user can pin / restore named checkpoints.

## In-scope this PR (MVP)

- **#108** Ed25519 signing of every history event. Key resolution via a
  pluggable `Signer` seam; the v0.3-4 default is a per-install key in
  `~/.aish/history.key` (0600). The secrets-engine-backed signer is
  deferred to v0.3-4.1.
- **#109** `aish checkpoint <name>` and `aish rollback <name>`. A
  checkpoint is a named, signed history event of kind `checkpoint`.
  Rollback restores every OpDelete in every event newer than the named
  checkpoint, walking back to (and including) the checkpoint marker.
- **#110** Modification snapshots. The destructive set grows: when `mv
  SRC DST` overwrites an existing DST, the prior DST bytes get an
  OpDelete snapshot row. The `>` / `>>` shell-redirect overwrite case
  is **out of scope this PR** — `shell/internal/parser` does not yet
  surface redirects; cross-partition work for `parser` is excluded by
  the spawn brief.
- **#111** Move / rename tracking. `mv SRC DST` records an `Affected`
  row with `Op == OpRename` carrying both source and target path. The
  source path's prior bytes are snapshotted (so `restore SRC` walks
  back). Reverse operations (`undo` of a rename) move bytes from DST
  back to SRC.
- **#113** `aish history list [N] | show <id> | search <query> | purge
  <selector>` built-in. Search uses **SQLite FTS5** (modernc.org/sqlite
  ships it; probed before commit). The FTS5 index is rebuilt by
  triggers on the events table.

## Deferred (filed as v0.3-4.1 follow-ups)

- **#112** Semantic search via embeddings. The shell has no embedding
  plugin contract yet (the inference plugin only exposes intent →
  invocation). Adding one is plugin-protocol surface (libs/proto/**),
  outside this partition. v0.3-4 lands keyword search as the realistic
  current floor; semantic search joins after the proto extension.
- Secrets-engine-backed signing. v0.3-3's plugin-side `secret` API is
  not yet wired into a "fetch the signing key" call. Stub-signer + the
  per-install key file is the safe v0.3-4 floor; the cross-engine
  integration lands in v0.3-4.1.
- Snapshot pruning policy (age / size / count). Listed in the spawn
  brief but not on any open issue; deferred to a v0.3-4.2 task.
- Cross-session history. Already works structurally — the SQLite store
  is process-shared via WAL — and the cross-session test gate proves
  it. No code change needed this PR.
- History export (`aish history export <range>` → JSONL). Optional in
  the spawn brief; deferred to a v0.3-4.2 task to keep the PR
  reviewable.

## Rationale (Alternatives Tables)

### Search backend

| Alternative                          | Pros                                                                                 | Cons                                                                                          | Verdict   |
|--------------------------------------|--------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------|-----------|
| **SQLite FTS5 virtual table**        | Ships with `modernc.org/sqlite` v1.50.1 (probed); zero new deps; same DB as events; phrase / prefix / boolean queries built-in. | Index rebuild cost on schema migration; locale-folding is generic Unicode (no stemming).      | **Chosen** |
| Ripgrep over a JSONL export          | Trivial implementation; familiar tooling.                                            | Requires an external binary (`rg`) on $PATH or vendoring; export step must run continuously.  | Rejected  |
| In-memory N-gram inverted index      | Pure Go, no SQLite virtual-table churn.                                              | Reload cost on every session; memory growth unbounded across multi-month history; reinvention. | Rejected  |

### Signing key resolution

| Alternative                                                       | Pros                                                                                                                       | Cons                                                                                       | Verdict   |
|-------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------|-----------|
| **Per-install file `~/.aish/history.key` (0600), generated lazily** | No cross-engine wiring needed this PR; deterministic per machine; preserves the "signed history" property end-to-end.       | Key lives on disk in cleartext (mitigated by 0600); rotation requires manual key deletion. | **Chosen for v0.3-4** |
| Secrets-engine-backed signer (v0.3-3 vault)                       | Reuses the Argon2id-protected vault; cross-engine integration is the long-term shape.                                       | Plugin-side `secret` API (task #106) not yet ready; would force a session-passphrase prompt before every history write — UX defect. | Deferred to v0.3-4.1 |
| Compiled-in dev key                                               | Zero on-disk surface.                                                                                                       | Every install would share a key — signing reduces to a checksum. Useless for tamper evidence. | Rejected  |

### Modification snapshots

| Alternative                                                                       | Pros                                                                                            | Cons                                                                                                   | Verdict   |
|-----------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------|-----------|
| **`mv` overwrites only — extend the destructive detector and snapshot DST when it exists** | Single-stage win: solves the most common modification path inside the parser's current surface. | Misses `>` and `>>` shell-redirect overwrite (parser does not yet surface them).                       | **Chosen** |
| Block on parser support for redirects, snapshot both                              | Complete coverage of the modification surface.                                                  | Parser is out of this partition (cross-team). Would block #110 indefinitely.                          | Rejected  |
| Filesystem-level inotify / fsevents observer                                      | Catches *every* modification regardless of shell tool.                                          | Massive scope creep; OS-specific; uncertain across remote filesystems; security-sensitive surface.    | Rejected  |

## Scope

### Files created

- `shell/internal/history/sign.go` — `Signer` interface, `FileSigner`
  (loads / creates `~/.aish/history.key`), `Verify` helper.
- `shell/internal/history/search.go` — FTS5 schema + `Search` / `List`
  / `Get` / `Purge` queries on `*Store`.
- `shell/internal/shell/builtin_history.go` — `history list / show /
  search / purge` dispatch.

### Files modified

- `shell/internal/history/event.go` — `OpRename`, `OpModify`,
  `KindCheckpoint` constants; `Event.Signature`, `Event.SignerID`,
  `Event.Name`, `Event.RenameTarget` fields.
- `shell/internal/history/schema.go` — `events.signature`,
  `events.signer_id`, `events.name`, `snapshots.rename_target`
  columns; FTS5 virtual table + triggers; migration that adds the
  new columns on existing DBs (`ALTER TABLE … ADD COLUMN`, idempotent
  via `PRAGMA table_info`).
- `shell/internal/history/store.go` — `Append` signs the event,
  persists signature columns; `Checkpoint(name)` writes a
  `KindCheckpoint` event; `List`, `Get`, `EventsSinceCheckpoint`
  query helpers.
- `shell/internal/history/detect.go` — `mv` joins the destructive
  set; `TargetPaths` returns `[SRC, DST]` for `mv`; new
  `RenameTargets(pl)` helper returns `(src, dst)` pairs.
- `shell/internal/history/snapshot.go` — `SnapshotMove(src, dst)`
  records a rename pair: snapshots SRC bytes, and (if DST exists)
  snapshots DST bytes too. Restore() learns OpRename.
- `shell/internal/history/interceptor.go` — `Before` branches on `mv`
  to call `SnapshotMove`; passes the active signer to `store.Append`.
- `shell/internal/shell/shell.go` — single dispatch branch for the
  `history` built-in; `Rollback(name)` wiring.

### Tests

- `shell/internal/history/sign_test.go` — sign / verify round-trip;
  tamper detection (mutated payload fails verify); missing-key
  bootstrap creates the file at 0600.
- `shell/internal/history/search_test.go` — FTS5 list / search /
  purge; phrase queries; substring fallback when FTS rejects the query.
- `shell/internal/history/store_test.go` — extend: checkpoint write +
  `EventsSinceCheckpoint`; schema migration from a pre-v0.3-4 DB.
- `shell/internal/history/snapshot_test.go` — extend: `SnapshotMove`
  captures DST bytes when DST exists; `Restore(OpRename)` puts bytes
  back at SRC.
- `shell/internal/history/detect_test.go` — extend: `mv` in
  destructive set; rename target extraction.
- `shell/internal/shell/builtin_history_test.go` — `history list`,
  `history show <id>`, `history search`, `history purge --before <ts>`.
- `shell/internal/shell/shell_acceptance_test.go` — extend: live smoke
  for cross-session history (two-shell invocation sequence).

## Approach (commit slices)

The brief warns "6 tasks doesn't mean 6 commits." Honest split:

1. **`feat(history): Ed25519 event signing + schema migration`** — sign.go,
   schema/store/event changes, sign_test. Carries the schema migration
   for the new columns. Foundation for everything else.
2. **`feat(history): mv detection + rename / modification snapshots`** —
   detect/snapshot/interceptor changes, OpRename, OpModify, the
   destructive set growth, snapshot_test + detect_test updates.
3. **`feat(history): FTS5-backed search + history built-in`** — search.go,
   builtin_history.go, shell.go dispatch wire-up, builtin_history_test.
4. **`feat(history): checkpoint + rollback`** — KindCheckpoint, Checkpoint
   writer, EventsSinceCheckpoint, the rollback path in shell.go +
   integration into the existing undo seam.
5. **`docs(history): v0.3-4 plan committed alongside changes`** — the plan
   file at `.artifacts/plans/v0.3-4.md` (this file). Committed last so
   the plan exists in main from the same SHA as the code that implements it.

If a slice needs to grow, it grows; the discipline is one logical
change per commit, not "exactly five commits."

## Risk assessment

- **Schema migration on a populated history.db.** Mitigation: the
  migration uses `ALTER TABLE … ADD COLUMN` with `PRAGMA table_info`
  probes; new columns default NULL / empty; existing payloads survive.
  Tested in store_test against a pre-v0.3-4 fixture DB.
- **FTS5 index churn.** Mitigation: rebuild-on-open is O(N) over all
  events; v0.3-4 histories are bounded by user activity (typically
  thousands of rows). A pruning policy is a v0.3-4.2 follow-up.
- **Signing key on disk.** Mitigation: 0600 permissions on creation;
  `~/.aish/history.key` documented; rotation is "delete + restart"
  with the explicit caveat that prior signatures will no longer verify
  against the new key (rebuild verifier UX is v0.3-4.1).
- **mv overwrite + cross-filesystem mv.** Mitigation: the snapshot
  branch runs before the actual `mv`, so cross-filesystem
  `EXDEV`-induced copy+delete still gets a pre-snapshot of both SRC
  and (if extant) DST.

## Dependencies

- `crypto/ed25519` (stdlib). No new direct deps.
- `modernc.org/sqlite v1.50.1` (already vendored). FTS5 confirmed
  available via a probe (`CREATE VIRTUAL TABLE t USING fts5(c)`
  succeeds).

## Backward compatibility

- Existing v0.1-4 history.db files migrate transparently on the next
  `aish` start. Old events get signature='', signer_id='', name=NULL
  — unsigned but readable. New events are signed.
- The `undo` and `restore` built-ins keep their v0.1-4 semantics. Old
  history events still undo correctly because the signature columns
  are additive.
- `history list/show/search/purge` is a new built-in — no clash.
