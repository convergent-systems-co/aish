# Plan: v1.0-5 — Windows Login Shell

> Bring aish's login-shell capabilities to Windows. v0.3-1 (PR #190)
> shipped the POSIX path — argv[0]-dash / `-l` / `--login` detection,
> `/etc/aish/aishrc` + `$HOME/.aish/aishrc.toml` sourcing, umask,
> `logout` / `exec`. Windows has none of the POSIX conventions — no
> `getty`-style dash-argv, no `/etc` system RC, no `$HOME` (it is
> `%USERPROFILE%`), and "default shell" is a registry hive, not
> `/etc/shells` + `chsh`. This PR delivers the **narrowly-scoped
> Windows-side login surface** needed to make aish viable as a
> Windows default shell.

**Source:** GOALS.md §"Epic v1.0-5 — Windows Login Shell"
**Epic (supervised):** #149
**Tasks (in scope this PR):** #150 (Windows console host integration —
the *login-mode* slice only), #151 (PTY/ConPTY — *RC-location slice
only*), #152 (registry changes for default shell).
**Tasks deferred this PR:** the bulk of #150 / #151 — ConPTY runtime,
console-host attachment, raw-mode toggles — belong to TL_WIN_OS
(`shell/internal/exec/**`). This PR owns the RC-file location split
and the registration helper only. Cross-TL handshake recorded below.
**Domain:** `domain:shell-runtime`
**Risk:** HIGH (per GOALS.md). Mitigated by scoping this PR to
**pure RC-path resolution + a PowerShell registration helper** —
no ConPTY, no console-host changes, no exec-pipeline rewrites. All
new Windows code is build-tagged `//go:build windows` and the
existing POSIX path is untouched at the source level (extended via
new build-tagged files, not refactored). Per `Code.md §11.3` this
is a pure extension, not a refactor.
**Milestone:** v1.0 — Windows Native (120 days)

## Cross-TL coordination

This batch is one of four parallel TLs working `feat/v1.0-*` worktrees:

| TL | Worktree | Owns |
|---|---|---|
| TL_WIN_BUILD | `aish-win-build` | `Makefile`, `.github/workflows/**` |
| TL_WIN_OS | `aish-win-os` | `shell/internal/exec/**` (ConPTY, console host) |
| TL_WIN_SECRETS | `aish-win-secrets` | `shell/internal/secrets/**` |
| **TL_WIN_LOGIN (this PR)** | `aish-win-login` | `shell/internal/shell/login_*.go`, `scripts/win/**` |

The registration helper script (`scripts/win/register-shell.ps1`)
*calls* the aish binary but does not depend on TL_WIN_BUILD's
Makefile changes — the script accepts an `-AishPath` parameter,
defaulting to `aish.exe` on `$env:PATH`.

The ConPTY work (#151's bulk) lands in TL_WIN_OS. This PR's #151
slice is limited to: documenting that RC-sourcing precedes any
console-host attachment, and ensuring `applyLoginEnvDefaults` does
not assume a POSIX-style `$PATH` shape on Windows (already true in
v0.3-1 via the `runtime.GOOS != "windows"` guard).

## Acceptance Criteria

### MVP-floor (shipping this PR)

- [ ] **Windows per-user RC location.** When `loginMode == true` on
      `GOOS=windows`, the shell sources
      `%APPDATA%\aish\aishrc.toml` (the Windows equivalent of
      `~/.aish/aishrc.toml`). Missing file = silent skip, matching
      v0.3-1 POSIX behavior.
- [ ] **Windows system-wide RC location.** When `loginMode == true`
      on `GOOS=windows`, the shell sources
      `%PROGRAMDATA%\aish\aishrc.toml` (the Windows equivalent of
      `/etc/aish/aishrc`) **before** the per-user file. Missing
      file = silent skip. User file wins on key collisions.
- [ ] **Build-tag split, not refactor.** A new file
      `login_paths_windows.go` (build tag `windows`) exposes
      `systemRCPath()` and `userRCRelPath(home string) string`
      functions that return the platform-correct paths. The
      existing `login.go` is split into `login_paths_posix.go`
      (build tag `!windows`) for the POSIX constants and a
      `login_common.go` (untagged) for shared logic. Behavior on
      POSIX is byte-identical to v0.3-1; this is verified by
      keeping every existing test green without modification.
- [ ] **`homeDir()` already handles `%USERPROFILE%`** (verified in
      `shell.go` — POSIX prefers `$HOME`, Windows falls through to
      `$USERPROFILE`). No change needed; the test suite gets a new
      Windows-tagged test confirming `%APPDATA%`-fallback when
      `$HOME` is unset.
- [ ] **PowerShell registration script
      (`scripts/win/register-shell.ps1`).** A PowerShell 5.1+
      compatible script that, when run as Administrator (HKLM
      paths) or as a user (HKCU paths), registers aish such that
      Windows can launch it as the user's default shell. Two
      modes:
      - `-Scope User` (default): writes to
        `HKCU:\Software\Microsoft\Command Processor\AutoRun` or
        sets up an `aish` profile entry for Windows Terminal —
        TBD per the §11.1 alternatives table below.
      - `-Scope Machine`: writes to
        `HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon`
        (the classic "user shell" replacement — risky, gated by
        an interactive confirmation prompt + a `-Force` switch).
      The script accepts `-AishPath` (default: `aish.exe` on
      `$env:PATH`), validates the binary exists, and writes a
      dated entry to `%LOCALAPPDATA%\aish\install.log`. The
      script is **idempotent** — re-running with the same args is
      a no-op. Reversal is documented inline (
      `-Unregister` switch).
- [ ] **Cross-compile gate.** `GOOS=windows GOARCH=amd64 go build
      ./shell/...` succeeds from macOS. Verified locally before
      the PR opens (no Windows runner needed for this slice).
- [ ] **Tests on macOS for Windows-tagged paths.** The
      `login_paths_windows_test.go` file is build-tagged
      `windows` so it does not run on macOS CI — but its logic is
      mirrored by a `TestUserRCRelPath_Windows` test in a
      cross-platform file that exercises a pure helper accepting
      `goos` as a parameter. Pure-function path resolution is
      testable on any host.

### Deferred (out of scope for this PR)

- [ ] **ConPTY runtime / console-host attach** — TL_WIN_OS, #150 +
      #151. The exec pipeline rewrite to spawn under ConPTY is
      strictly outside this lane.
- [ ] **Windows Terminal profile JSON generator** — possible
      follow-up. The registration script can write a Terminal
      profile fragment, but parsing and merging
      `settings.json` is fiddly and we defer it until users ask.
- [ ] **MSI / winget installer** — TL_WIN_BUILD owns packaging.
      The PS1 script is the install-step floor; an MSI is the
      ceiling.
- [ ] **Windows umask equivalent.** Windows has no umask;
      `applyUmask` is already a no-op on Windows (verified in
      `login_umask_windows.go`). No change.

## Alternatives Tables

### §1 — Per-user RC location on Windows

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **`%APPDATA%\aish\aishrc.toml`** (Roaming) | Conventional for app config on Windows; roams with user profile in AD environments; matches PowerShell's `$PROFILE` location pattern | Roaming behavior may surprise users on locked-down corporate boxes (read-only roaming) | **Chosen** |
| `%LOCALAPPDATA%\aish\aishrc.toml` (Local) | Never roams; predictable on corporate boxes | Doesn't follow the user across machines; less conventional for *config* (Local is for cache/state) | Rejected — config belongs in Roaming |
| `%USERPROFILE%\.aish\aishrc.toml` | Direct POSIX-mirror; same path shape | Dotfile convention is foreign on Windows; Explorer hides it; not where Windows tools look | Rejected — wrong-platform convention |
| `%USERPROFILE%\aish\aishrc.toml` | Visible to users; no dotfile | Pollutes the profile root; not where apps store config on Windows | Rejected |

### §2 — Shell-registration mechanism

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **PowerShell script (`register-shell.ps1`)** writing to HKCU/HKLM Winlogon + Terminal profile | Standard Windows admin pattern; user-runnable without admin for `-Scope User`; PowerShell is on every supported Windows host | Requires execution-policy adjustment on some systems (script signs as a follow-up) | **Chosen** |
| `winget`-published post-install hook | Future-friendly; one-command install | Requires aish to ship via winget first — TL_WIN_BUILD territory; doesn't help users on pre-winget Windows 10 | Rejected for this PR — re-evaluate when TL_WIN_BUILD ships an MSI |
| Manual `reg.exe` instructions in README | Zero-script-surface; minimal trust required | Hostile to users; high-error; no idempotency | Rejected — `install-shells` POSIX prints instructions, but Windows registry is too fiddly for paste-and-pray |
| Go-side registration via `golang.org/x/sys/windows/registry` from a `aish setup-default-shell` subcommand | Single-binary install; testable in Go | Requires elevation; risks bricking the user's login if aish.exe is broken at next login | Rejected — too dangerous; PS1 with `-Unregister` rollback is safer |

## Approach

1. **TDD.** Write `login_paths_test.go` (untagged) that exercises a
   pure-function `userRCRelPathForGoos(goos string, home string)
   string` returning the platform-correct absolute path. Tests
   pass `"windows"` and `"linux"` and assert the expected output.
2. Split `login.go`:
   - Move `systemRCPath` constant + `userRCRelPath` constant into
     a new `login_paths_posix.go` (build tag `!windows`).
   - Create `login_paths_windows.go` (build tag `windows`) with
     the equivalent Windows constants resolved at runtime via
     `os.Getenv("APPDATA")` / `os.Getenv("PROGRAMDATA")`.
   - Refactor `loadRCFiles` to call a small platform-tagged
     helper for the per-user path. The system-wide path stays a
     simple `systemRCPath` reference; on Windows it becomes a
     runtime lookup (function, not constant, because `%PROGRAMDATA%`
     can vary).
3. Add a Windows-tagged test
   `login_paths_windows_test.go` covering the `%APPDATA%`
   fallback (`$HOME` unset, `%USERPROFILE%` set, `%APPDATA%` set).
4. Add `scripts/win/register-shell.ps1` per §2 above with
   `-Scope`, `-AishPath`, `-Unregister`, `-Force`, `-WhatIf`
   parameters. The script is the floor; integration testing
   requires a Windows VM (UTM smoke per §Verification).
5. Run `make ci` (POSIX side green).
6. Run `GOOS=windows GOARCH=amd64 go build ./shell/...` and
   confirm clean cross-compile.
7. PR.

## Testing Strategy

- **Unit, POSIX.** Existing `login_test.go` continues to pass
  unchanged. This is the regression gate that proves the
  build-tag split is behavior-preserving on macOS/Linux.
- **Unit, Windows-tagged.** New
  `login_paths_windows_test.go` (`//go:build windows`) covers
  `%APPDATA%` path resolution. Runs on Windows CI only — but
  the cross-compile gate (`go build`) verifies it parses on
  every push.
- **Unit, pure-function.** A new
  `login_paths_test.go` (untagged) exercises a
  `userRCPathForGoos(goos, home, appdata string) string`
  helper that takes the OS as a parameter, so the resolution
  logic is testable from any host without build tags.
- **PS1 script.** Manually testable via UTM Windows VM (post-merge
  smoke). The script has no unit-test framework here; the
  `-WhatIf` switch lets a reviewer dry-run it without writes.
- **Cross-compile gate.** `GOOS=windows GOARCH=amd64 go build
  ./shell/...` MUST be clean. Added as a one-liner in the PR
  body's verification section.

## Risk Assessment

- **Registry writes brick login.** Mitigated by:
  defaulting `-Scope` to `User` (HKCU, not HKLM); requiring
  `-Force` for `Machine` scope; supporting `-Unregister` and
  `-WhatIf`; logging every change to
  `%LOCALAPPDATA%\aish\install.log`.
- **Roaming-profile lockdown.** Some corporate AD environments
  make `%APPDATA%` read-only or quota-limited. Mitigated by
  documenting `%LOCALAPPDATA%` as a fallback in the RC schema
  doc — but the default stays `%APPDATA%` because that's the
  Windows convention.
- **Build-tag drift.** Two files now define `systemRCPath` /
  `userRCRelPath` — risk of divergence. Mitigated by keeping
  both as small constants/helpers, both covered by the
  pure-function test (`userRCPathForGoos`), and the existing
  `login_test.go` continuing to pass.
- **Cross-TL collision.** TL_WIN_BUILD may add a `scripts/win/`
  directory for build artifacts. Mitigated by namespacing this
  PR's script as `scripts/win/register-shell.ps1` (named for
  its function, not its TL).

## UTM smoke (post-merge)

To verify end-to-end on Windows:

1. Cross-compile `aish.exe` from macOS:
   `GOOS=windows GOARCH=amd64 go build -o aish.exe ./shell/cmd/aish`
2. Boot a Windows 11 UTM VM. Copy `aish.exe` and
   `scripts/win/register-shell.ps1`.
3. From an elevated PowerShell session:
   `.\register-shell.ps1 -AishPath C:\Tools\aish.exe -WhatIf`
   — confirm dry-run output matches expectations.
4. Re-run without `-WhatIf`. Log to
   `%LOCALAPPDATA%\aish\install.log` exists with one entry.
5. Create `%APPDATA%\aish\aishrc.toml` with
   `[env] AISH_SMOKE = "windows-1"`.
6. Launch aish with `-l` from a new PowerShell session; confirm
   `echo $env:AISH_SMOKE` returns `windows-1` inside aish.
7. `.\register-shell.ps1 -Unregister` and confirm log entry +
   registry cleanup.
