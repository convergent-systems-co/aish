## Intent

Make aish behave as a POSIX-style job-controlling shell on Unix:

- `cmd &` runs in the background; the prompt returns immediately with
  `[<jobid>] <pgid>` printed to stderr.
- `jobs` lists the active job table.
- `fg [%n]` resumes a job in the foreground; `bg [%n]` continues a
  stopped job in the background (SIGCONT).
- Ctrl-Z (SIGTSTP) suspends the foreground job and returns control.
- Ctrl-C (SIGINT) is forwarded to the foreground process group; aish
  itself ignores SIGINT so the REPL survives.
- On child completion, aish prints `[<jobid>]+  Done   <command>`
  on the next prompt.

Windows is a stub. JobObject-based job control is v1.0 work; we ship
a clear stderr message and `jobs`/`fg`/`bg` exit 1 on Windows.

## Acceptance Criteria

### #84 — Process group management (backbone)

- [ ] Every external child started by the shell on Unix is placed in
      its own process group via `SysProcAttr.Setpgid: true, Pgid: 0`.
      Pipeline children share the first stage's pgid (multi-stage jobs
      receive Ctrl-C and SIGTSTP as a unit).
- [ ] Foreground jobs receive the controlling TTY via
      `unix.IoctlSetInt(fd, unix.TIOCSPGRP, pgid)` before `Wait`, and
      aish reclaims the TTY (`TIOCSPGRP` back to the shell's own pgid)
      on every exit path including panic.
- [ ] The REPL goroutine ignores `SIGTTIN`/`SIGTTOU` for the shell
      process so backgrounded jobs that touch the TTY don't suspend the
      shell itself.
- [ ] `SIGINT`, `SIGQUIT`, `SIGTSTP` are ignored by the REPL goroutine
      (children receive them via the foreground pgrp).
- [ ] Windows: `SysProcAttr` setup is a no-op (separate file already
      stubs the unix-only fields out via build tags); job control
      surfaces a polite error.

### #83 — `jobs` / `fg` / `bg` built-ins

- [ ] `cmd &` parses (trailing-`&` parser extension) into a `Pipeline`
      with `Background: true`. The runtime spawns it as a background
      job and prints `[<jobid>] <pgid>` to stderr, then returns to the
      prompt without waiting.
- [ ] `jobs` lists current jobs in `[<jobid>]<flag>  <Status>   <cmd>`
      form. `<flag>` is `+` for the current job, `-` for the previous,
      space otherwise. Statuses are `Running`, `Stopped`, `Done`.
- [ ] `fg` (no arg) resumes the most recent backgrounded or stopped
      job in the foreground. `fg %<n>` selects a specific job. Missing
      job → stderr + exit 1.
- [ ] `bg [%<n>]` sends SIGCONT to a stopped job, keeps it in the
      background, marks status `Running`.
- [ ] Ctrl-Z while a foreground job runs:
      (1) shell sends `SIGTSTP` to the fg pgrp (kernel does this for
      us via the controlling-TTY routing) →
      (2) job becomes `Stopped` in the table →
      (3) shell reclaims the TTY → (4) prompt returns.
- [ ] On `SIGCHLD`, the shell's signal handler updates the job
      table (Done / Stopped / Continued). On the next prompt render,
      newly-Done jobs are reported as `[<jobid>]+  Done   <cmd>` and
      then evicted from the table.
- [ ] Verified live on macOS (per task brief) — synthetic test
      processes use `exec.LookPath("sleep")`, never `os.Getpid()`.

## Rationale + Alternatives

### Reaping strategy: SIGCHLD vs polling

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **SIGCHLD handler + `waitpid(-1, WNOHANG\|WUNTRACED\|WCONTINUED)`** | True POSIX shape. Zero CPU when idle. Captures stops/continues. | Requires a long-lived signal-notification goroutine + a mutex-guarded job table. | **Chosen** |
| Periodic polling of `Process.Wait` per known pid | Pure stdlib. No signal-handler complexity. | `Wait` is blocking; non-blocking variant needs syscall anyway. Misses stop/continue transitions. Doesn't scale. | Rejected |

### Process-group setup: `Setpgid` vs `Setsid`

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **`Setpgid: true, Pgid: 0` per first stage; later stages join via `Pgid: <first>`** | Shell remains in the foreground session and keeps the controlling TTY. Standard bash/zsh approach. | One extra step (set child's pgid post-fork) — `os/exec` exposes both `Setpgid` and `Pgid` so it's one struct literal. | **Chosen** |
| `Setsid: true` per child | Simpler to set up; child detached cleanly. | Detaches from controlling TTY entirely — child can't receive Ctrl-C from the parent's TTY. Wrong shape for fg jobs. | Rejected |

### Foreground-job wait: synchronous wait vs async + state machine

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Synchronous `os/exec.Cmd.Wait` on the REPL goroutine, with the SIGCHLD goroutine updating the job table in parallel** | Keeps `runExternal` semantically identical to today — the REPL still blocks until the foreground job finishes. Ctrl-Z works because the kernel SIGSTOPs the child; our `Wait` returns with `WaitStatus.Stopped() == true`, we update the table, return to the prompt. | The wait blocks; if SIGCHLD fires for the same pid in the parallel handler, we must avoid double-reaping (the handler skips fg-job pids — they're owned by the REPL goroutine). | **Chosen** |
| Async wait + REPL state machine (REPL never blocks; foreground completion is a callback that re-emits the prompt) | Pure event-loop shape. | Major rewrite of `Run()`; subtle prompt-redraw bugs; opt-out from the byte-by-byte stdin invariant (#167). Not justified for v0.3-1. | Rejected |

### Parser surgery: trailing `&` token vs grammar rewrite

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Tokenizer recognises a final unquoted `&` and the parser sets `Pipeline.Background = true`** | Surgical. Touches `tokenize()` once + one new `Pipeline` field. Existing tests untouched. | Mid-line `&` (`a & b`) is not supported — POSIX bash treats it as a sequence separator; we reject it as a syntax error and leave it for a future PR. | **Chosen** |
| Full bash-style grammar (`&` as statement separator, `;`, `&&`, `‖`) | Future-proof. | Significant grammar rewrite. Out of scope per the task brief ("surgical"). | Rejected |

## Threat Model — Signals and TTY

- **REPL deadlock if shell doesn't ignore SIGTTOU.** When a
  backgrounded child writes to the TTY, the kernel sends SIGTTOU to
  its pgrp. If the shell shares that pgrp (we don't, but defense in
  depth) it would suspend. Mitigation: explicit `signal.Ignore(SIGTTOU,
  SIGTTIN, SIGTSTP, SIGINT, SIGQUIT)` in the shell's process.
- **TTY ownership leak.** If a panic between `TakeTTY(fg_pgid)` and
  `ReleaseTTY(shell_pgid)` happens, the next prompt would be invisible
  (shell isn't fg pgrp). Mitigation: `defer ReleaseTTY` on every fg
  path; verify via the existing PTY raw-mode test pattern.
- **Double-reap race.** Foreground `cmd.Wait` and the SIGCHLD reaper
  both call wait*(). The reaper skips pids known to be foreground
  via `JobTable.IsForeground(pid)`; foreground wait happens
  synchronously in the REPL goroutine.
- **Test-runner suspension.** A `syscall.Kill(os.Getpid(), SIGTSTP)`
  in a test would suspend the `go test` runner itself. Tests
  ONLY operate on `exec.LookPath("sleep")` child processes spawned
  by the test, never on the test process.
- **Windows.** Job control is unsupported; `jobs`/`fg`/`bg` print
  a clear stderr message and return exit 1. Background pipelines
  on Windows surface the same error and run as foreground.

## Verification

1. `make ci` — exits 0 across all modules.
2. `GOOS=windows GOARCH=amd64 go build ./shell/...` — cross-compile
   gate is clean.
3. Live smoke on macOS (must be done manually because CI has no PTY):
   - `sleep 30 &` → prompt returns immediately; `jobs` shows
     `[1]  Running   sleep 30`.
   - `fg %1` → foregrounds, completes, exit captured into `$?`.
   - `sleep 30` then Ctrl-Z → `[1]+  Stopped   sleep 30`; `bg %1`
     resumes (status flips to Running); `wait` not needed because
     SIGCHLD updates the table.
   - Two simultaneous `&` jobs → `jobs` lists both with correct
     `+`/`-` markers.
4. Unit tests:
   - `jobs.JobTable` state transitions (Add → Stopped → Done; Add
     → Done; Add two, current/previous tracking).
   - `parser` trailing `&` cases.
   - `exec.applySysProcAttr` sets `Setpgid` on a recorded `*Cmd`
     stub.
5. Adversarial pass:
   - What if `sleep 30 &` is run with no controlling TTY (stdin
     piped)? Expected: still runs in background; the TTY-ownership
     dance is skipped (no fg job ever installed). Test: pipe a
     single-line script and assert exit 0.
   - What if the user runs `sleep 100 &` then `exit`? Expected:
     shell `Close()` kills surviving background jobs and reaps.
